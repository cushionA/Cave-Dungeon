# 01 - データコンテナとソースジェネレーター

## 1. 概要

本プロジェクトでは、キャラクターに関するすべてのデータを **SoA (Structure of Arrays)** パターンで一元管理する。
従来の Unity 開発では各 GameObject にコンポーネントを付与し `GetComponent<T>()` で取得するのが一般的だが、
本設計ではすべてのキャラクター情報を **`GameManager.Data`** (= `SoACharaDataDic`) という単一のデータコンテナに格納し、
**objectHash をキーとした O(1) アクセス**で取得する。

この設計により以下の利点を得る。

- `GetComponent` の完全排除によるパフォーマンス向上
- データの局所性向上によるキャッシュヒット率の改善
- すべてのキャラクターデータへの統一的なアクセスパターン
- SourceGenerator による型安全なコンテナコードの自動生成

すべてのキャラクター情報は以下の統合構造体でコンテナ管理される:
CharacterVitals, CombatStats, CharacterFlags, MoveParams, EquipmentStatus, CharacterColdLog, DamageScoreEntry[], CharacterStatusEffects 等。

> **構造体統合の詳細**: [07_AI判定システム再設計](./07_AI判定システム再設計.md) セクション6 参照

なお、SoAコンテナに格納するのは **値型データ（unmanaged構造体）** のみである。
コンテナが管理する managed 型（BaseCharacter 参照）を通じて、キャラクターコントローラーや被弾管理クラス等の
MonoBehaviour 参照にもアクセスできる。これにより GetComponent を完全に排除しつつ、
コンテナに直接持たせる必要のない参照型データにもハッシュベースでアクセス可能となる。

## 2. アクセスパターン

### 2.1 GetComponent 排除の設計

Unity 標準の `GetComponent<T>()` はフレームごとに呼び出すとパフォーマンスのボトルネックになる。
本設計では、各オブジェクトの **ハッシュコード (int)** をキーとして `GameManager.Data` からデータを直接参照する。

```csharp
// ─────────────────────────────────────────────
// [従来] GetComponent地獄
// ─────────────────────────────────────────────
var health = target.GetComponent<Health>();
var attack = target.GetComponent<AttackPower>();
// → 毎回型検索が走り、キャッシュ効率も悪い

// ─────────────────────────────────────────────
// [本設計] hashだけでO(1)アクセス
// ─────────────────────────────────────────────
int hash = target.GetHashCode();
ref var vitals      = ref GameManager.Data.GetVitals(hash);       // HP/MP/位置等
ref var combatStats = ref GameManager.Data.GetCombatStats(hash);  // 攻撃力+防御力
ref var flags       = ref GameManager.Data.GetFlags(hash);        // 全状態フラグ
BaseCharacter chara = GameManager.Data.GetManaged(hash);          // MonoBehaviour参照
// → ハッシュテーブルの1回の参照でデータに直接到達
// → ref返しにより値コピーも発生しない
// → managed型経由でコントローラー等にもアクセス可能（GetComponent不要）
```

重要な点として、**コードが objectHash を持ってさえいれば**、`GameManager.Data` 経由で任意のデータに即座にアクセスできる。
`GetComponent` は一切不要である。

### 2.2 各コンポーネントからのアクセス

どのコンポーネントからでも、対象の `objectHash` さえ知っていれば `GameManager.Data` 経由でデータにアクセスできる。

| コンポーネント | hash の取得方法 | 用途例 |
|---|---|---|
| **BaseCharacter** | `self.ObjectHash` でアクセス | 自身のステータス参照・更新 |
| **WeaponHolder** | `owner.ObjectHash` でアクセス | 武器所有者のステータス参照 |
| **DamageReceiver** | 被弾元の hash でアクセス | ダメージ計算時に攻撃元情報を参照 |
| **AI / Behavior** | 対象の hash でアクセス | 敵味方のステータス評価 |
| **任意の外部コード** | hash さえ知っていればアクセス可能 | UI表示、デバッグ、エフェクト等 |

### 2.3 アクセスフロー図

```
任意のコード
    │ objectHash を持っている
    ▼
GameManager.Data (= SoACharaDataDic)
    │ Hash Table で index を解決
    ▼
T[index] → データ直接参照 (ref返し)
```

すべてのアクセスは以下の2ステップに集約される。

1. **ハッシュ → インデックス変換**: `Dictionary<int,int>` で `objectHash` から配列インデックスを解決（逆引き `int[] _indexToHash` も保持）
2. **インデックス → データ参照**: 各 `T[]` 配列の同一インデックスからデータを `ref` で返す

**重要**: BaseCharacter は自身のデータのコピーを持たない。常に SoACharaDataDic を経由してアクセスする。これにより：
- データの一元管理
- メモリ使用量の最適化

## 3. SourceGenerator の仕組み

コンテナクラスのボイラープレートコード (Add / Remove / Get / Dispose 等) を手書きすると、
構造体を追加するたびに大量の定型コードの修正が必要になる。
本プロジェクトでは **SourceGenerator** を用いてこれらを自動生成する。

### 3.1 ContainerSettingAttribute

`[ContainerSetting]` 属性をコンテナクラスの `partial class` 宣言に付与することで、
SourceGenerator がそのクラスに必要なメソッド群を自動生成する。

```csharp
/// <summary>
/// この属性をpartialクラスに付けると、SourceGeneratorが
/// SoAコンテナのコードを自動生成する。
/// </summary>
[AttributeUsage(AttributeTargets.Class)]
public class ContainerSettingAttribute : Attribute
{
    /// <summary>管理する unmanaged 構造体の型リスト</summary>
    public Type[] StructTypes { get; }

    /// <summary>管理する managed クラスの型（MonoBehaviour等）</summary>
    public Type ManagedType { get; }

    public ContainerSettingAttribute(Type managedType, params Type[] structTypes)
    {
        ManagedType = managedType;
        StructTypes = structTypes;
    }
}
```

### 3.2 コンテナクラスの宣言例

開発者が記述するのは以下の `partial class` 宣言のみ。
`[ContainerSetting]` に managed 型と各構造体型を指定する。

> **実装上の設計補足（PR #29 対応）**
> 現行パッケージ `com.zabuton.container-pack` の `ContainerSettingAttribute` は `classType[j].IsClass` チェックがあり、**インターフェース型を classType に指定できない**。
> また `SoACharaDataDic` は `Game.Core.asmdef` に属し、`BaseCharacter` は `GameCode.asmdef`（Runtime）に属するため、Core → Runtime の直接参照ができない。
> この2つの制約のため、`classType: new[] { typeof(BaseCharacter) }` は記述できない。
>
> **現実的な解決**: Core 側に `public abstract class ManagedCharacter : MonoBehaviour` を定義し（`ObjectHash` と `Damageable` 抽象メンバーのみ持つ最小契約）、`BaseCharacter : ManagedCharacter` で継承する。
> SoACharaDataDic は `classType: new[] { typeof(ManagedCharacter) }` で自動生成され、`GetManaged(hash) → ManagedCharacter` が公開される。
> Runtime 側のダメージパイプラインは `GameManager.Data.GetManaged(hash)?.Damageable` で `IDamageable` を取得する。
>
> なお「Core に MonoBehaviour を置かない」という原則には緊張があるが、`ManagedCharacter` は SoA 管理契約のための最小抽象なので、例外的に許容する。

```csharp
/// <summary>
/// キャラクターデータのSoAコンテナ。
/// SourceGeneratorが各構造体に対応するUnsafeListとアクセサを生成する。
/// </summary>
[ContainerSetting(
    typeof(BaseCharacter),           // managed型（MonoBehaviour参照。コントローラー・被弾管理等への参照を保持）
    typeof(CharacterVitals),         // HP/MP/スタミナ/アーマー/位置（旧 BaseInfo + StaminaInfo を統合）
    typeof(CombatStats),             // 攻撃力+防御力+盾データ（旧 AtkStatus + DefStatus を統合）
    typeof(CharacterFlags),          // 全状態フラグ ulong（旧 StateInfo + SolidData + RecognitionData を統合）
    typeof(MoveParams),              // 移動パラメータ（旧 MoveStatus を改名）
    typeof(EquipmentStatus),         // 装備由来ステータス合計値
    typeof(CharacterColdLog),        // 判定クールダウン（AI用）
    typeof(CharacterStatusEffects)   // 状態異常蓄積・発症スロット
)]
public partial class SoACharaDataDic : IDisposable
{
    // 実装では managed T[] 配列ベース:
    // - CharacterVitals[] _vitals;
    // - CombatStats[] _combatStats;
    // - CharacterFlags[] _flags;
    // - MoveParams[] _moveParams;
    // - EquipmentStatus[] _equipmentStatus;
    // - CharacterStatusEffects[] _statusEffects;
    // - Dictionary<int,int> _hashToIndex + int[] _indexToHash（双方向マッピング）
    // - Add(), Remove(), TryGetValue(), Dispose()
    // - GetVitals(hash), GetCombatStats(hash) 等のref返しアクセサ
    //
    // --- managed型経由でアクセスできるもの ---
    // BaseCharacter が保持する参照（GetComponent不要）:
    // - CharacterController（物理・移動制御）
    // - DamageReceiver（被弾管理）
    // - AbilityHolder（能力コンポーネント群）
    // - その他外部からアクセス頻度の高いMonoBehaviour
    // → GameManager.Data.GetManaged(hash) で BaseCharacter を取得し、そこから参照
}
```

### 3.3 自動生成されるコード

SourceGenerator は `[ContainerSetting]` に指定された各構造体に対し、以下のコードを自動生成する。

#### フィールド

```csharp
// === SourceGenerator 自動生成コード ===
public partial class SoACharaDataDic : IDisposable
{
    // ---- フィールド（managed T[] 配列ベース） ----
    private CharacterVitals[] _vitals;                // HP/MP/スタミナ/アーマー/位置
    private CombatStats[] _combatStats;               // 攻撃力+防御力+盾+クリティカル
    private CharacterFlags[] _flags;                  // 全状態フラグ（ulong）
    private MoveParams[] _moveParams;                 // 移動パラメータ
    private EquipmentStatus[] _equipmentStatus;       // 装備由来ステータス
    private CharacterStatusEffects[] _statusEffects;  // 状態異常蓄積・アクティブ効果

    // ハッシュ ↔ インデックスの双方向マッピング
    private Dictionary<int, int> _hashToIndex;        // hash → dense index
    private int[] _indexToHash;                        // dense index → hash（swap-back用）
    private int _count;
```

#### Add メソッド

```csharp
    // ---- 追加 ----
    public int Add(int hashCode, BaseCharacter managed,
                   CharacterBaseInfo baseInfo, SolidData solid,
                   CharacterAtkStatus atk, CharacterDefStatus def,
                   CharacterStateInfo state, MoveStatus move,
                   CharacterColdLog coldLog, RecognitionData recognition,
                   StaminaInfo stamina)
    {
        int index = _count;
        _characterBaseInfo.Add(baseInfo);
        _solidData.Add(solid);
        _characterAtkStatus.Add(atk);
        _characterDefStatus.Add(def);
        _characterStateInfo.Add(state);
        _moveStatus.Add(move);
        _characterColdLog.Add(coldLog);
        _recognitionData.Add(recognition);
        _staminaInfo.Add(stamina);
        _managedArray[index] = managed;
        RegisterHash(hashCode, index);
        _count++;
        return index;
    }
```

#### Remove メソッド (swap-back 方式)

```csharp
    // ---- 削除（スワップバック方式） ----
    public void Remove(int hashCode)
    {
        if (!TryGetIndex(hashCode, out int index)) return;

        int lastIndex = _count - 1;
        if (index != lastIndex)
        {
            // 末尾要素を削除位置にスワップ
            _characterBaseInfo[index] = _characterBaseInfo[lastIndex];
            _solidData[index] = _solidData[lastIndex];
            _characterAtkStatus[index] = _characterAtkStatus[lastIndex];
            _characterDefStatus[index] = _characterDefStatus[lastIndex];
            _characterStateInfo[index] = _characterStateInfo[lastIndex];
            _moveStatus[index] = _moveStatus[lastIndex];
            _characterColdLog[index] = _characterColdLog[lastIndex];
            _recognitionData[index] = _recognitionData[lastIndex];
            _staminaInfo[index] = _staminaInfo[lastIndex];
            _managedArray[index] = _managedArray[lastIndex];
            UpdateHash(_managedArray[index].GetHashCode(), index);
        }
        // 末尾を除去
        _characterBaseInfo.Length--;
        _solidData.Length--;
        _characterAtkStatus.Length--;
        _characterDefStatus.Length--;
        _characterStateInfo.Length--;
        _moveStatus.Length--;
        _characterColdLog.Length--;
        _recognitionData.Length--;
        _staminaInfo.Length--;
        _count--;
    }
```

## 4. ハッシュテーブル構造

`SoACharaDataDic` の内部構造は以下の通り。
ハッシュテーブルで `objectHash` からインデックスを解決し、
すべての `UnsafeList<T>` は同一インデックスで対応するデータを保持する。

```
objectHash (int)
    │
    ▼
┌──────────────────────────────────────────────────┐
│ Hash Table (素数サイズ、ロードファクター 0.75)    │
│                                                  │
│  Bucket[0] → Entry(hash=..., index=2)            │
│  Bucket[1] → Empty                               │
│  Bucket[2] → Entry(hash=..., index=0) → chain    │
│  ...                                             │
│  Bucket[N] → Entry(hash=..., index=5)            │
└──────────────────────────────────────────────────┘
        │ index を使って各 UnsafeList にアクセス
        ▼
┌──────────────────────────────────────────────────┐
│ UnsafeList<CharacterBaseInfo>  [0][1][2][3]...   │
│ UnsafeList<SolidData>          [0][1][2][3]...   │
│ UnsafeList<CharacterAtkStatus> [0][1][2][3]...   │
│ UnsafeList<CharacterDefStatus> [0][1][2][3]...   │
│ UnsafeList<CharacterStateInfo> [0][1][2][3]...   │
│ UnsafeList<MoveStatus>         [0][1][2][3]...   │
│ UnsafeList<CharacterColdLog>   [0][1][2][3]...   │
│ UnsafeList<RecognitionData>    [0][1][2][3]...   │
│ UnsafeList<StaminaInfo>        [0][1][2][3]...   │
│ UnsafeList<PersonalityData>    [0][1][2][3]...   │  ← AI拡張
│ UnsafeList<AnimationStateData> [0][1][2][3]...   │  ← AI拡張
│ UnsafeList<DeliberationBuffer> [0][1][2][3]...   │  ← AI拡張
│ UnsafeList<EquipmentStatus>        [0][1][2]...  │  ← 装備拡張
│ UnsafeList<CharacterStatusEffects> [0][1][2]...  │  ← 状態異常拡張
│ BaseCharacter[]                [0][1][2][3]...   │
└──────────────────────────────────────────────────┘

※ 各配列の同一インデックスが同一キャラクターのデータ
```

**SoA パターンのメリット**: 同種のデータが連続メモリに配置されるため、
特定のフィールドだけを走査する処理 (例: 全キャラの HP を一括チェック) で
CPU キャッシュラインの利用効率が最大化される。

## 5. 登録フロー

キャラクターがシーンに登場してからデータコンテナに登録されるまでの流れ。

```
CharacterInfo (ScriptableObject) をエディタで設定
    │
    ▼
BaseCharacter.Initialize()
    │ ScriptableObject から各構造体を生成
    │
    ├── CharacterBaseInfo  ← CharacterInfo.baseData から生成
    ├── SolidData          ← CharacterInfo.solidData から生成
    ├── MoveStatus         ← CharacterInfo.moveData から生成
    ├── CharacterAtkStatus ← CharacterInfo.baseData から計算
    ├── CharacterDefStatus ← CharacterInfo.baseData から計算
    ├── StaminaInfo        ← CharacterInfo.staminaData から生成
    ├── CharacterStateInfo ← 初期状態で生成
    ├── CharacterColdLog   ← 初期値で生成
    ├── RecognitionData    ← 空で生成
    ├── PersonalityData    ← LLM出力 or デフォルト値(0.5)で生成
    ├── AnimationStateData ← Neutral / 0 で生成
    └── DeliberationBuffer ← 難易度に応じた初期遅延で生成
    │
    ▼
GameManager.CharacterAdd(BaseCharacter chara)
    │ 各構造体を初期値から構築
    ▼
SoACharaDataDic.Add(hash, managed, baseInfo, solid, atk, def, ...)
    │ ハッシュテーブルに hash → index を登録
    │ 各 UnsafeList<T> に構造体を追加
    ▼
return index
    │ BaseCharacter.charIndex に保存
    ▼
登録完了 → 以降 hash で O(1) アクセス可能
```

### 削除フロー

キャラクターが破棄されるとき:

```
BaseCharacter.OnDestroy()
    │
    ▼
GameManager.CharacterRemove(hash)
    │
    ▼
SoACharaDataDic.Remove(hash)
    │ swap-back方式で削除
    │ 末尾要素を削除位置に移動
    │ ハッシュマッピングを更新
    ▼
削除完了
```

## 6. メモリ管理

### 6.1 アロケーション戦略

| 用途 | Allocator | 寿命 |
|---|---|---|
| SoACharaDataDic の各リスト | `Persistent` | ゲーム全体 |
| センサー出力一時データ | `stackalloc` | メソッドスコープ |

### 6.2 容量管理

- **初期容量**: 推定最大キャラ数 (例: 64)
- **自動拡張**: ロードファクター 0.75 超過で素数サイズにリサイズ
- **縮小**: しない (頻繁なリアロケーション回避)

### 6.3 Dispose チェーン

`Allocator.Persistent` で確保したネイティブメモリは明示的に解放する必要がある。
**GameManager.OnDestroy** を起点とした Dispose チェーンで解放を保証する。

```
GameManager.OnDestroy()
    │
    ├─→ Data.Dispose()                    // SoACharaDataDic
    │       ├─→ _characterBaseInfo.Dispose()
    │       ├─→ _solidData.Dispose()
    │       ├─→ _characterAtkStatus.Dispose()
    │       ├─→ _characterDefStatus.Dispose()
    │       ├─→ _characterStateInfo.Dispose()
    │       ├─→ _moveStatus.Dispose()
    │       ├─→ _characterColdLog.Dispose()
    │       ├─→ _recognitionData.Dispose()
    │       ├─→ _staminaInfo.Dispose()
    │       ├─→ _personalityData.Dispose()       // AI拡張
    │       ├─→ _animationStateData.Dispose()    // AI拡張
    │       ├─→ _deliberationBuffer.Dispose()    // AI拡張
    │       └─→ _hashToIndex 関連の解放
    │
    └─→ relationMap.Dispose()
```

### 6.4 メモリリーク防止

- **エディタ再生終了時**: `OnDestroy` が確実に呼ばれるよう `OnApplicationQuit` でもバックアップ解放を行う
- **例外発生時**: Dispose を `try-finally` で保護する
- **確認方法**: Unity の Native Memory Leak Detection を有効化し、未解放のネイティブコンテナを検出する

## 7. 新しい構造体の追加手順

新しいデータ型をコンテナに追加する場合、以下の手順で行う。

### 手順 1: unmanaged な構造体を定義する

```csharp
// 例: バフ情報を追加したい場合
[StructLayout(LayoutKind.Sequential)]
public struct BuffInfo
{
    public int buffId;
    public float remainTime;
    public float effectValue;
}
```

**注意**: 構造体は `unmanaged` 型でなければならない (参照型フィールドを含まないこと)。

### 手順 2: ContainerSetting に型を追加する

```csharp
[ContainerSetting(
    typeof(BaseCharacter),
    typeof(CharacterBaseInfo),
    typeof(SolidData),
    typeof(CharacterAtkStatus),
    typeof(CharacterDefStatus),
    typeof(CharacterStateInfo),
    typeof(MoveStatus),
    typeof(CharacterColdLog),
    typeof(RecognitionData),
    typeof(StaminaInfo),
    typeof(BuffInfo)             // ← 追加
)]
public partial class SoACharaDataDic : IDisposable { }
```

### 手順 3: リビルドする

SourceGenerator が自動的に以下を生成する。

- `UnsafeList<BuffInfo> _buffInfo` フィールド
- `Add` メソッドに `BuffInfo` パラメータ追加
- `Remove` メソッドに swap-back 処理追加
- `GetBuffInfo(int hashCode)` ハッシュベースアクセサ生成
- `GetBuffInfoByIndex(int index)` インデックス直接アクセサ生成
- `Deconstruct` に `out UnsafeList<BuffInfo>` 追加
- `Dispose` メソッドに解放処理追加

### 手順 4: 登録処理を更新する

`GameManager.CharacterAdd` で新しい構造体の初期値を渡す。

```csharp
var buffInfo = new BuffInfo
{
    buffId      = 0,
    remainTime  = 0f,
    effectValue = 0f,
};

int index = Data.Add(
    chara.ObjectHash,
    chara,          // managed
    baseInfo,
    solid,
    atkStatus,
    defStatus,
    stateInfo,
    moveStatus,
    coldLog,
    recognition,
    staminaInfo,
    buffInfo        // ← 追加
);
```

### 手順 5: 必要な箇所からアクセスする

```csharp
// ハッシュベースアクセス
ref var buff = ref GameManager.Data.GetBuffInfo(hash);
buff.effectValue = 1.5f;
buff.remainTime  = 10.0f;

// インデックス直接アクセス
ref var buff = ref GameManager.Data.GetBuffInfoByIndex(index);
```

### チェックリスト

新しい構造体を追加する際の確認事項:

- [ ] 構造体は `unmanaged` 型か (参照型フィールドを含まないか)
- [ ] `[ContainerSetting]` の型リストに追加したか
- [ ] ビルドが通るか (SourceGenerator のエラーがないか)
- [ ] `GameManager.CharacterAdd` で初期値を渡しているか
- [ ] 既存のテストが壊れていないか
- [ ] 必要に応じて新しい構造体用のテストを追加したか

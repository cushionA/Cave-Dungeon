# 07 - AI判定システム再設計

## 1. 概要

本ドキュメントは、旧AIシステム（05_AIシステム.md）の判定ロジックを再設計したものである。
旧設計の3種の判定enum（ActTriggerCondition / TargetSelectCondition / MoveSelectCondition, 計41バリアント）を
統一された **AICondition** システム（12種）に統合し、判定フローを明確な階層構造に整理する。

### 1.1 旧設計の問題点

| 問題 | 詳細 |
|------|------|
| **条件enumの重複** | 3つの判定enum（トリガー/ターゲット/行動）に7割同じ条件が重複定義されていた |
| **型安全性の欠如** | `judgeLowerValue` / `judgeUpperValue` が int で、条件ごとに CharacterBelong, BitableBool, RecognizeObjectType 等を手動キャスト |
| **エディタコードの肥大化** | 3構造体 × 同一ShowIf/HideIfロジックのコピペ。エディタコードが構造体定義の3倍以上 |
| **ヘイトシステムの複雑性** | teamHate / personalHate の NativeHashMap が必要だったが、2Dメトロヴァニアでは不要 |
| **AND/OR結合の不自然さ** | judgeLowerValue に BitableBool をキャストしてAND/OR判定を表現 |

### 1.2 新設計の方針

| 方針 | 詳細 |
|------|------|
| **判定フローの階層化** | ターゲット切替 → 行動切替 → デフォルト行動 の明確な3層構造 |
| **条件enumの統一** | 12種の `AIConditionType` + `CompareOp` で全条件を表現 |
| **ヘイト廃止** | 累積ダメージスコア（DamageScore）で代替。ヘイト上昇状態は倍率で表現 |
| **行動は番号で直接指定** | AIRule の `actionIndex` が ActionData 配列のインデックスを指す |
| **TargetFilter は維持** | 旧設計の TargetFilter（陣営・特徴・属性・距離でのフィルタリング）はそのまま活用 |

## 2. 判定フロー

### 2.1 3層判定アーキテクチャ

```
┌─────────────────────────────────────────────────────┐
│  第1層: ターゲット切替判定                            │
│  targetRules[] を優先度順に評価                       │
│  → マッチした場合: ターゲットを切替                    │
│  → マッチなし: 現ターゲットを維持                      │
├─────────────────────────────────────────────────────┤
│  第2層: 行動切替判定（ターゲット固定状態で）           │
│  actionRules[] を優先度順に評価                       │
│  → マッチした場合: その行動を実行                      │
│  → マッチなし: デフォルト行動                          │
├─────────────────────────────────────────────────────┤
│  第3層: 行動実行                                      │
│  actions[index] を実行（条件不要、常に何かする）       │
└─────────────────────────────────────────────────────┘
```

### 2.2 旧設計との比較

| 旧設計 | 新設計 |
|--------|--------|
| トリガー判定 → モード変更/ターゲット変更/特別行動（混在） | 第1層: ターゲット切替のみ |
| ターゲット判定 → ヘイト/条件/固定ターゲット（混在） | 第1層の結果: TargetSortKey で選択 |
| 行動判定 → 行動選択（ターゲット未確定のまま） | 第2層: ターゲット確定後に行動選択 |
| デフォルト行動なし → 条件全不一致で棒立ち | 第3層: 必ずデフォルト行動が存在 |

### 2.3 メリット

- **ターゲットが確定しているから行動条件が「相対的」に書ける**（ターゲットとの距離、位置関係）
- **条件全スカでもデフォルト行動がある** → 棒立ちしない
- **DamageScore はターゲット切替の判断材料** であって行動を決めるものではない

## 3. データ構造

### 3.1 AIConditionType（条件の種類）

```csharp
/// <summary>
/// AI判定で使用する条件の種類。
/// TargetFilter との組み合わせで旧設計の41バリアントを完全に代替する。
/// </summary>
public enum AIConditionType : byte
{
    None = 0,

    // --- 数値比較系 ---
    Count,            // フィルター該当キャラ数
    HpRatio,          // HP割合（%）
    MpRatio,          // MP割合（%）
    StaminaRatio,     // スタミナ割合（%）
    ArmorRatio,       // アーマー割合（%）

    // --- 空間系 ---
    Distance,         // ターゲットとの距離
    NearbyFaction,    // 周囲の特定陣営キャラ数
    ProjectileNear,   // 飛び道具が一定距離内にあるか
    ObjectNearby,     // 指定オブジェクト/地形が周囲にあるか（RecognizeObjectType）

    // --- スコア・状態系 ---
    DamageScore,      // 累積ダメージスコア（閾値判定）
    EventFired,       // イベントフラグ（BrainEventFlagType）
    SelfActState,     // 自分の行動状態（ActState）
}
```

#### 旧条件との対応表

| 旧 ActTriggerCondition | 新 AIConditionType | 備考 |
|---|---|---|
| 一定の対象が一定数いる時 | `Count` + TargetFilter | フィルターで対象を限定 |
| HPが一定割合の対象がいる時 | `HpRatio` + TargetFilter | |
| MPが一定割合の対象がいる時 | `MpRatio` + TargetFilter | |
| 対象のキャラの周囲に同陣営が一定以上密集 | `NearbyFaction` | |
| 対象のキャラの周囲に同陣営が一定以下しかいない | `NearbyFaction` + CompareOp.Less | |
| 周囲に指定のオブジェクトや地形がある時 | `ObjectNearby` | |
| 対象が一定数の敵に狙われている時 | `Count` + TargetFilter(狙われ数) | |
| 対象のキャラの一定距離以内に飛び道具がある時 | `ProjectileNear` | |
| 一定のイベントが発生した時 | `EventFired` | |

| 旧 TargetSelectCondition（28種） | 新設計での対応 |
|---|---|
| ヘイト | **廃止** → `DamageScore` で代替 |
| HP割合 / HP | `TargetSortKey.HpRatio` / `TargetSortKey.HpValue` |
| 各属性別攻撃力（×7） | `TargetSortKey.AttackPower` + `Element` 指定 |
| 各属性別防御力（×7） | `TargetSortKey.DefensePower` + `Element` 指定 |
| 自分 / プレイヤー / シスター | `TargetSortKey.Self` / `Player` / `Sister` |

### 3.2 CompareOp（比較演算子）

```csharp
/// <summary>
/// AICondition で使用する比較演算子。
/// 数値比較とビットフラグ判定の両方をカバーする。
/// </summary>
public enum CompareOp : byte
{
    Less,          // 値 < operandA
    LessEqual,     // 値 <= operandA
    Equal,         // 値 == operandA
    GreaterEqual,  // 値 >= operandA
    Greater,       // 値 > operandA
    NotEqual,      // 値 != operandA
    InRange,       // operandA <= 値 <= operandB
    HasFlag,       // (値 & operandA) == operandA  (AND判定)
    HasAny,        // (値 & operandA) != 0          (OR判定)
}
```

### 3.3 AICondition（条件1つ）

```csharp
/// <summary>
/// AI判定の最小単位。条件タイプ + 比較演算子 + オペランド + フィルターで構成。
/// 旧設計の TriggerJudgeData / ActJudgeData / TargetJudgeData を統一した構造体。
/// </summary>
[Serializable]
public struct AICondition
{
    /// <summary>条件の種類</summary>
    public AIConditionType type;

    /// <summary>比較演算子</summary>
    public CompareOp op;

    /// <summary>
    /// 比較値A（閾値下限、ビットフラグ等）。
    /// 条件タイプに応じて意味が変わる:
    /// - Count/HpRatio/MpRatio等: 数値閾値
    /// - ObjectNearby: RecognizeObjectType のビットフラグ
    /// - EventFired: BrainEventFlagType のビットフラグ
    /// - SelfActState: ActState のビットフラグ
    /// </summary>
    public int operandA;

    /// <summary>
    /// 比較値B（InRange の上限値等）。
    /// InRange 以外では未使用。
    /// </summary>
    public int operandB;

    /// <summary>チェック対象を絞り込むフィルター</summary>
    public TargetFilter filter;
}
```

### 3.4 AIRule（ルール = 条件配列のAND結合）

```csharp
/// <summary>
/// AIの判定ルール。複数の AICondition を AND 結合し、
/// 全条件を満たした場合に actionIndex で指定された行動/ターゲット選択を実行する。
/// 複数の AIRule は配列として優先度順に並べ、先にマッチしたルールが採用される（OR結合）。
/// </summary>
[Serializable]
public struct AIRule
{
    /// <summary>条件配列（全条件を満たす = AND結合）</summary>
    public AICondition[] conditions;

    /// <summary>
    /// 結果インデックス。使用場所によって意味が変わる:
    /// - targetRules 内: targetSelects[actionIndex] を使用
    /// - actionRules 内: actions[actionIndex] を使用
    /// </summary>
    public int actionIndex;

    // NOTE: weightやprobabilityは実装時に決定
}
```

### 3.5 AITargetSelect（ターゲット選択方法）

```csharp
/// <summary>
/// ターゲットの選択方法を定義する。
/// 旧設計の TargetSelectCondition（28種）を、ソートキー + 属性指定 + 反転フラグで代替。
/// </summary>
public enum TargetSortKey : byte
{
    Distance,        // ターゲットとの距離
    HpRatio,         // HP割合
    HpValue,         // HP実値
    AttackPower,     // 攻撃力（elementFilter で属性指定）
    DefensePower,    // 防御力（elementFilter で属性指定）
    TargetingCount,  // 狙われている数
    LastAttacker,    // 最後に自分を攻撃した相手
    DamageScore,     // 累積ダメージスコアが最大の相手
    Self,            // 自分自身（固定）
    Player,          // プレイヤー（固定）
    Sister,          // シスター（固定）
}

/// <summary>
/// ターゲット選択の設定データ。
/// TargetFilter で候補を絞り込み、TargetSortKey で最適な1体を選ぶ。
/// </summary>
[Serializable]
public struct AITargetSelect
{
    /// <summary>ソート基準</summary>
    public TargetSortKey sortKey;

    /// <summary>属性指定（AttackPower/DefensePower 用）</summary>
    public Element elementFilter;

    /// <summary>降順=最大値を選ぶ、昇順=最小値を選ぶ</summary>
    public bool isDescending;

    /// <summary>候補を絞り込むフィルター</summary>
    public TargetFilter filter;
}
```

### 3.6 ActionSlot（行動スロット）

```csharp
/// <summary>
/// 行動の実行パターン分類。
/// ActionSlot の execType で指定し、実行時のディスパッチ先を決定する。
/// </summary>
public enum ActionExecType : byte
{
    Attack,     // AttackMotionData参照。ヒットボックス・motionValue・コンボ管理
    Cast,       // 詠唱→発動モーション→ProjectileSystem（弾丸飛翔）
    Instant,    // アニメ1回再生して完了（ワープ、回避、アイテム使用等）
    Sustained,  // 開始→毎フレームTick→終了条件で終了（移動、ガード等）
    Broadcast,  // イベント送信（味方指示、挑発等）
}

/// <summary>
/// 全キャラAI共通の行動定義。
/// AIRule の actionIndex がこの配列のインデックスを指す。
///
/// 「処理パターン（ActionExecType）× 具体的な行動名（ActionType）」の組み合わせで
/// あらゆる行動を表現する。ActionType enum は行動追加に伴い拡張されるため、
/// ここでは定義を省略する。
/// </summary>
[Serializable]
public struct ActionSlot
{
    /// <summary>エディタ表示用の名前</summary>
    public string displayName;

    /// <summary>実行パターン（Attack/Cast/Instant/Sustained/Broadcast）</summary>
    public ActionExecType execType;

    /// <summary>
    /// ターゲットは第1層（targetRules）で確定済み。ActionSlotにターゲット上書きはない。
    /// 連携アクション（CoopAction）のみコンボ段ごとのターゲットをCoopActionManager側で独自管理する。
    /// </summary>

    /// <summary>
    /// 行動固有のID。execType に応じて意味が変わる:
    /// - Attack: AttackMotionData のインデックス
    /// - Cast: MagicDefinition ID
    /// - Sustained: パトロールルート ID 等
    /// - Instant/Broadcast: 行動固有データ ID
    /// </summary>
    public int paramId;

    /// <summary>
    /// 汎用パラメータ。execType × paramId の組み合わせで意味が変わる:
    /// - Sustained: 持続時間（秒）。0 = 無期限（次の行動判定で切り替わるまで持続）
    ///   NPCカスタムAIではプレイヤーがこの値を自由に設定可能
    /// - Instant: WarpBehind→オフセット距離等
    /// - Broadcast: 効果持続時間（秒）
    /// - Attack/Cast: 未使用（定義側にデータがある）
    /// </summary>
    public float paramValue;

}
```

### 3.7 AIMode（モードデータ）

```csharp
/// <summary>
/// AIの1つのモードを定義する。
/// BrainStatus が複数の AIMode を持ち、モード変更によって行動パターンが切り替わる。
/// </summary>
[Serializable]
public struct AIMode
{
    /// <summary>モード名（エディタ表示用）</summary>
    public string modeName;

    /// <summary>

    /// マッチしたルールの actionIndex → actions[i] で行動を決定。

    public AITargetSelect[] targetSelects;

    /// <summary>行動スロットの配列（actionRules から参照される）</summary>
    public ActionSlot[] actions;

    /// <summary>行動ルールが全不一致の場合のデフォルト行動インデックス</summary>
    public int defaultActionIndex;

    /// <summary>判定間隔（秒）。x: ターゲット判定間隔, y: 行動判定間隔</summary>
    public Vector2 judgeInterval;
}
```

## 4. 累積ダメージスコア（ヘイト代替）

### 4.1 概要

旧設計のヘイトシステム（teamHate / personalHate の NativeHashMap）を廃止し、
**被ダメージ記録 + 倍率** のシンプルな仕組みで代替する。

### 4.2 設計方針

- 各キャラクターが固定長の被ダメージログを持つ（画面内の敵味方数は限られるため 4-8 枠で十分）
- ダメージを受けた時にスコアを加算（ヘイト上昇状態なら倍率適用）
- 時間減衰により「最近攻撃してこない相手は忘れる」を表現
- ターゲット選択の `TargetSortKey.DamageScore` で「最もスコアが高い相手」を選択
- `AIConditionType.DamageScore` で「スコアが閾値以上の相手がいるか」を判定

### 4.3 データ構造

```csharp
/// <summary>
/// 累積ダメージスコアの1エントリ。
/// 各キャラクターが固定長配列（4-8枠）で保持する。
/// </summary>
public struct DamageScoreEntry
{
    /// <summary>攻撃者のハッシュコード</summary>
    public int attackerHash;

    /// <summary>
    /// 累積ダメージスコア。
    /// 実ダメージ × 倍率（ヘイト上昇状態なら1.2倍等）で加算される。
    /// </summary>
    public float score;

    /// <summary>最終更新時刻（時間減衰の基準）</summary>
    public float lastUpdateTime;
}
```

### 4.4 スコア計算

```
ダメージ受領時:
  score += actualDamage * (攻撃者がヘイト上昇状態 ? hateMultiplier : 1.0f)
  lastUpdateTime = currentTime

毎判定時（減衰）:
  elapsed = currentTime - lastUpdateTime
  score *= decayRate ^ elapsed  // 例: 0.95^秒数

条件評価:
  AIConditionType.DamageScore → 指定閾値以上のスコアを持つ攻撃者が存在するか

ターゲット選択:
  TargetSortKey.DamageScore → スコア最大の攻撃者を選択
```

### 4.5 ゲームプレイ上の効果

- **ヘイト上昇状態のキャラ**が攻撃すると、実ダメージ以上にスコアが溜まる → 自然とターゲットを引きつける
- **閾値方式**により、盾を構えているだけではターゲットを取れない → 「ちゃんと攻撃して敵意を向けさせる」プレイに意味が出る
- **時間減衰**により、長時間攻撃しない相手からは興味を失う
- ヘイトテーブル（NativeHashMap）の管理コストがゼロになる

## 5. TargetFilter（変更なし・継続使用）

旧設計の `TargetFilter` はそのまま継続使用する。
条件チェックの対象を絞り込む役割であり、新設計でも全く同じ用途で使用される。

```csharp
/// <summary>
/// AI判定の対象を絞り込むフィルター。
/// 陣営・特徴・属性・距離等の組み合わせで候補をフィルタリングする。
/// 各フィルターのAND/OR判定は FilterMode のビットフラグで制御する。
/// </summary>
[Serializable]
[StructLayout(LayoutKind.Sequential)]
public struct TargetFilter : IEquatable<TargetFilter>
{
    /// <summary>各フィルターのAND/OR判定切替</summary>
    [Flags]
    public enum FilterMode : byte
    {
        BelongAnd       = 1 << 0,  // 陣営フィルターをAND判定
        FeatureAnd      = 1 << 1,  // 特徴フィルターをAND判定
        ActStateAnd     = 1 << 2,  // 行動状態フィルターをAND判定
        EventAnd        = 1 << 3,  // イベントフィルターをAND判定
        WeakPointAnd    = 1 << 4,  // 弱点属性フィルターをAND判定
        UseElementAnd   = 1 << 5,  // 使用属性フィルターをAND判定
        IsSelf          = 1 << 6,  // 自分を対象に含める
        IsPlayer        = 1 << 7,  // プレイヤーを対象に含める
    }

    public FilterMode filterFlags;         // AND/OR切替ビットフラグ
    public CharacterBelong belong;         // 陣営（複数指定可）
    public CharacterFeature feature;       // 特徴（複数指定可）
    public ActState actState;              // 行動状態（複数指定可）
    public BrainEventFlagType eventFlags;  // イベント（複数指定可）
    public Element weakPoint;              // 弱点属性（複数指定可）
    public Element useElement;             // 使用属性（複数指定可）
    public float2 distanceRange;           // 距離範囲（min, max）
}
```

## 6. SoA データ構造の統合

### 6.1 旧構造体 → 新構造体マッピング

旧設計の9構造体を5構造体に統合する。

| 旧構造体 | 新構造体 | 変更内容 |
|---|---|---|
| CharacterBaseInfo (22B) | **CharacterVitals** | HP/MP/スタミナ/アーマー/位置を統合 |
| StaminaInfo | **CharacterVitals** | ↑に統合 |
| CharacterAtkStatus (32B) | **CombatStats** | 攻撃+防御+盾を1構造体に |
| CharacterDefStatus (40B) | **CombatStats** | ↑に統合 |
| CharacterStateInfo (17B) | **CharacterFlags** (ulong) | 陣営/行動/効果/イベントをビットパック |
| SolidData (20B) | **CharacterFlags** + ScriptableObject | ビットフラグは flags に、固定データは ScriptableObject 側に残す |
| RecognitionData | **CharacterFlags** + クエリシステム | 認識ビットは flags に、nearbyCount はクエリで取得 |
| MoveStatus (16B) | **MoveParams** | 変更なし |
| EquipmentStatus | **EquipmentStatus** | 変更なし |

### 6.2 CharacterVitals

```csharp
/// <summary>
/// キャラクターの生存情報。HP/MP/スタミナ/アーマー/位置を統合。
/// 旧 CharacterBaseInfo + StaminaInfo を統合。
/// </summary>
[Serializable]
[StructLayout(LayoutKind.Sequential)]
public struct CharacterVitals
{
    public short maxHp;
    public short currentHp;
    public short maxMp;
    public short currentMp;
    public short maxStamina;
    public short currentStamina;
    public short maxArmor;
    public short currentArmor;
    public byte hpRatio;          // HP% (0-100)
    public byte mpRatio;          // MP% (0-100)
    public float staminaRecoveryRate;
    public float staminaRecoveryDelay;
    public float2 position;       // 現在位置

    public void UpdateRatios()
    {
        hpRatio = (byte)(maxHp > 0 ? currentHp * 100 / maxHp : 0);
        mpRatio = (byte)(maxMp > 0 ? currentMp * 100 / maxMp : 0);
    }
}
```

### 6.3 CombatStats

```csharp
/// <summary>
/// 戦闘ステータス。攻撃力+防御力+盾データを統合。
/// 旧 CharacterAtkStatus + CharacterDefStatus を統合。
/// </summary>
[Serializable]
[StructLayout(LayoutKind.Sequential)]
public struct CombatStats
{
    public ElementalStatus attack;    // 属性別攻撃力
    public ElementalStatus defense;   // 属性別防御力
```

```csharp
/// <summary>
/// キャラクターの状態フラグ。ulong にビットパックして1命令で複合条件チェック可能。
/// 旧 CharacterStateInfo + SolidData(一部) + RecognitionData(一部) を統合。
///
/// ビットレイアウト (64bit):
/// [0-2]   CharacterBelong (3bit)
/// [3-8]   ActState (6bit)
/// [9-17]  CharacterFeature (9bit)
/// [18-23] SpecialEffect + 状態異常 (6bit)
/// [24-35] RecognizeObjectType (12bit)
/// [36-43] BrainEventFlagType (8bit)
/// [44-51] CharacterConditions (8bit)
/// [52-63] 予約 (12bit)
/// </summary>
[StructLayout(LayoutKind.Sequential)]
public struct CharacterFlags
{
    public ulong flags;

    // ビット操作ヘルパー
    public CharacterBelong Belong => (CharacterBelong)((flags >> 0) & 0x7);
    public ActState State => (ActState)((flags >> 3) & 0x3F);
    public CharacterFeature Feature => (CharacterFeature)((flags >> 9) & 0x1FF);
    public RecognizeObjectType RecognizedObjects => (RecognizeObjectType)((flags >> 24) & 0xFFF);
    public BrainEventFlagType Events => (BrainEventFlagType)((flags >> 36) & 0xFF);

    // フラグ判定（1命令）
    public bool HasAll(ulong mask) => (flags & mask) == mask;
    public bool HasAny(ulong mask) => (flags & mask) != 0;
}
```

## 7. AIBrain 判定フロー（新設計）

### 7.1 Evaluate() の擬似コード

```csharp
public void Tick(float deltaTime, float nowTime)
{
    // 0. 判定タイマー更新（SoA 外のローカル float）
    _targetJudgeTimer -= deltaTime;
    _actionJudgeTimer -= deltaTime;

    AIMode currentMode = brainStatus.modes[_nowMode];

    // 1. ターゲット切替判定
    if (_targetJudgeTimer <= 0f)
    {
        _targetJudgeTimer = currentMode.judgeInterval.x;

        foreach (var rule in currentMode.targetRules)
        {
            if (EvaluateConditions(rule.conditions) && RollProbability(rule.probability))
            {
                AITargetSelect select = currentMode.targetSelects[rule.actionIndex];
                currentTarget = SelectTarget(select);
                break; // 先にマッチしたルールを採用
            }
        }
    }

    // 2. 行動切替判定
    if (_actionJudgeTimer <= 0f)
    {
        _actionJudgeTimer = currentMode.judgeInterval.y;

        int selectedAction = currentMode.defaultActionIndex; // デフォルト

        foreach (var rule in currentMode.actionRules)
        {
            if (EvaluateConditions(rule.conditions) && RollProbability(rule.probability))
            {
                selectedAction = rule.actionIndex;
                break;
            }
        }

        // 3. 行動実行
        ActionExecutor.Execute(currentMode.actions[selectedAction], objectHash, currentTarget);
    }
}
```

### 7.2 EvaluateConditions() の擬似コード

```csharp
/// <summary>
/// AICondition 配列を全て AND で評価する。
/// 全条件が true を返せばルール成立。
/// </summary>
private bool EvaluateConditions(AICondition[] conditions)
{
    foreach (var cond in conditions)
    {
        float value = GetConditionValue(cond);
        if (!Compare(value, cond.op, cond.operandA, cond.operandB))
        {
            return false; // 1つでも不成立なら即 false
        }
    }
    return true;
}

/// <summary>
/// 条件タイプに応じて評価値を取得する。
/// </summary>
private float GetConditionValue(AICondition cond)
{
    switch (cond.type)
    {
        case AIConditionType.Count:
            return CountFilteredTargets(cond.filter);
        case AIConditionType.HpRatio:
            return GetFilteredHpRatio(cond.filter);
        case AIConditionType.Distance:
            return GetDistanceToTarget();
        case AIConditionType.DamageScore:
            return GetMaxDamageScore(cond.filter);
        case AIConditionType.NearbyFaction:
            return CountNearbyFaction(cond.filter);
        case AIConditionType.ObjectNearby:
            return HasNearbyObject(cond.operandA); // operandA = RecognizeObjectType
        case AIConditionType.EventFired:
            return HasEvent(cond.operandA); // operandA = BrainEventFlagType
        // ... 他の条件タイプ
        default:
            return 0;
    }
}
```

## 9. パフォーマンス考慮

### 9.1 メモリ効率

| 項目 | 旧設計 | 新設計 |
|------|--------|--------|
| 条件構造体の種類 | 3種 + CoolTimeData | 1種（AICondition） |
| SoA構造体数 | 9+ | 7 (Vitals/CombatStats/Flags/MoveParams/Equipment/StatusEffects/AnimationState) |
| 判定クールダウン管理 | CharacterColdLog (SoA) | JudgmentLoop ローカル float タイマー（SoA外） |
| CharacterFlags | 4構造体に分散 | ulong 1つ（8byte） |
| ヘイトテーブル | NativeHashMap × 2 | DamageScoreEntry[4-8] per character（DamageScoreTracker 別コンテナ） |

### 9.2 キャッシュ効率

- `CharacterFlags` が ulong 1つなので、フィルタリング時のメモリアクセスが最小
- `(flags & mask) != 0` の1命令で複合条件チェック可能
- TargetFilter のフィルタリングも CharacterFlags のビット演算で高速化

### 9.3 判定回数の削減

- ターゲット切替と行動切替の判定間隔を独立設定可能
- ターゲットが安定している間はターゲット判定をスキップ可能
- デフォルト行動があるため、行動判定の条件数を最小限に抑えられる

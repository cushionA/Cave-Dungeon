## 1. 概要

本システムは **MonoBehaviour ベースのルール駆動 AI** であり、全キャラクターの AI 判定をメインスレッドで逐次実行するアーキテクチャである。

| 要素 | 役割 |
|---|---|
| **AIManager** | AI 判定の統括、イベント配信、チーム関係管理を担当するシングルトン |
| **GameManager.Data** | 全ランタイムデータの所有者。AIManager は `GameManager.Data` 経由でデータにアクセスする |
| **AIBrain** | MonoBehaviour コンポーネント。キャラクターごとに AI 判定を実行する |
| **SensorSystem** | 各キャラクターの知覚・認識を担当。TriggerSensor2D / RangeSensor2D で周囲を探査し RecognitionData を生成する |
| **BrainStatus** | ScriptableObject ベースの AI 設定データ。モード・条件・行動をデータ駆動で定義する |

### 設計原則

- **データは GameManager.Data が所有する** --- AIManager はデータコンテナを持たず、`GameManager.Data` を通じて SoA 配列にアクセスする
- **objectHash による一意識別** --- 全キャラクターを `GameObject.GetHashCode()` で識別し、SoACharaDataDic のキーとして使用する
- **SoA（Structure of Arrays）** --- キャッシュ効率最大化のため、キャラクターデータはフィールドごとの配列として管理する
- **unmanaged 構造体で SoA 互換** --- SoACharaDataDic に格納するデータは unmanaged 構造体で構成し、GC を回避する

## 2. AIManager

### 2.1 役割

`AIManager` は `MonoBehaviour` ベースのシングルトンであり、以下の責務を持つ。

| 責務 | 詳細 |
|---|---|
| **AI 判定の統括** | 毎フレーム `UpdateAI()` を呼び出し、各キャラクターの `AIBrain.Evaluate()` をメインスレッドで逐次実行する |
| **イベント配信** | `BrainEventContainer` を管理し、キャラクター間のイベント（大ダメージ付与、回復使用など）を仲介する |
| **チーム関係管理** | `relationMap`（int[]）でチーム間の敵対ビットを保持する |
| **~~ヘイト追跡~~** | **廃止** → 累積ダメージスコア（DamageScoreEntry）で代替。詳細は [07_AI判定システム再設計](./07_AI判定システム再設計.md) 参照 |
| **キャラクター登録・削除** | `CharacterAdd()` / `CharacterDead()` で SoACharaDataDic を同期する |

> **重要**: データコンテナ（`SoACharaDataDic`）の所有権は `GameManager` にある。AIManager は `GameManager.Data` を通じてアクセスし、自身がデータのライフサイクルを管理しない。

### 2.2 主要フィールド

```csharp
public class AIManager : MonoBehaviour, IDisposable
{
    public static AIManager instance;                  // シングルトン

    // GameManager.Data 経由でアクセスされるデータ
    public SoACharaDataDic characterDataDictionary;    // SoA キャラデータ辞書
    public int[] relationMap;                             // チーム間敵対ビット
    // ヘイトシステム廃止 → DamageScoreEntry（SoACharaDataDic内）で代替
    public List<BrainEventContainer> eventContainer;      // イベントキュー
    public CharacterStatusList brainStatusList;         // AI設定マスタ(ScriptableObject)

    // AIBrain コンポーネントの管理配列
    private AIBrain[] managedBrains;                    // 登録済みキャラクターの AIBrain 参照
    private int characterCount;                         // 現在の登録キャラクター数
}
```

### 2.3 UpdateAI() フロー

毎フレーム `Update()` から呼び出される AI 判定のメインフローである。

```
┌─ UpdateAI() ─────────────────────────────────────────────────────┐
│                                                                   │
│  1. 登録済み AIBrain を逐次実行                                   │
│     ┌─────────────────────────────────────────────────────────┐   │
│     │ for (int i = 0; i < characterCount; i++)                │   │
│     │ {                                                       │   │
│     │     managedBrains[i].Evaluate(nowTime);                 │   │
│     │ }                                                       │   │
│     └─────────────────────────────────────────────────────────┘   │
│                                                                   │
│  2. 各 AIBrain.Evaluate() が判定結果を直接                        │
│     BaseCharacter に反映する                                      │
│                                                                   │
└───────────────────────────────────────────────────────────────────┘
```

コード上の実装:

```csharp
private void UpdateAI()
{
    float nowTime = Time.time;

    for (int i = 0; i < characterCount; i++)
    {
        var brain = managedBrains[i];
        brain.Evaluate(nowTime);
    }
}
```

```csharp
public void CharacterAdd(BrainStatus status, BaseController addCharacter)
```

## 3. AIBrain (MonoBehaviour)

### 3.1 クラス定義

```csharp
public class AIBrain : MonoBehaviour
{
    private int objectHash;                // このキャラクターの一意識別子
    private BrainStatus brainStatus;       // AI設定（ScriptableObject）
    private BaseController controller;     // 行動実行先
}
```

AIBrain は各キャラクターの GameObject にアタッチされる MonoBehaviour コンポーネントであり、`Evaluate()` メソッドで AI 判定を実行する。

### 3.2 入力データ (GameManager.Data 経由)

AIBrain は `GameManager.Data` の `SoACharaDataDic` から `objectHash` をキーとしてデータを取得する。

| データ | 取得方法 | 内容 |
|---|---|---|
| `CharacterVitals` | `GameManager.Data.GetVitals(objectHash)` | HP/MP/スタミナ/アーマー/位置（旧 CharacterBaseInfo + StaminaInfo を統合） |
| `CombatStats` | `GameManager.Data.GetCombatStats(objectHash)` | 属性別攻撃力+防御力+盾データ（旧 AtkStatus + DefStatus を統合） |
| `CharacterFlags` | `GameManager.Data.GetFlags(objectHash)` | 全状態フラグ ulong（旧 StateInfo + SolidData + RecognitionData を統合） |
| `MoveParams` | `GameManager.Data.GetMoveParams(objectHash)` | 移動速度、歩行速度、ダッシュ速度、ジャンプ高さ |
| `DamageScoreEntry[]` | `GameManager.Data.GetDamageScores(objectHash)` | 累積ダメージスコア（ヘイト代替） |
| `relationMap` | `AIManager.instance.relationMap` | チーム間の敵対ビットマップ |
| `brainStatus` | `BrainStatus` (ScriptableObject) | AI 設定（AIMode 配列: ターゲットルール + 行動ルール） |
| `nowTime` | `Evaluate()` 引数 | 現在時刻 |

> **構造体統合の詳細**: [07_AI判定システム再設計](./07_AI判定システム再設計.md) セクション6 参照

### 3.3 出力データ

AIBrain は判定結果の `MovementInfo` を直接 `BaseCharacter` に適用する（中間バッファを経由しない）。

| フィールド | 型 | 内容 |
|---|---|---|
| `resultData` | `MovementInfo` | ターゲット変更、行動番号、移動指示などの判定結果 |

### 3.4 Evaluate() の判定ロジック

`Evaluate(float nowTime)` は以下の順序で判定を実行する。

```
Evaluate(nowTime)
│
├─ 1. 基本情報の取得
│   ├─ nowMode        ← JudgmentLoop 内部状態
│   ├─ passTime       ← JudgmentLoop の _targetJudgeTimer / _actionJudgeTimer
│   │                   (毎 Tick で減算、0 以下で判定実行→リセット。SoA 外のローカル float)
│   └─ judgeIntervals ← AIMode.judgeInterval (ターゲット/行動 別)
│
├─ 2. ターゲット切替判定 (_targetJudgeTimer <= 0 の場合)
│   ├─ currentMode.targetRules[] を優先度順に評価
│   ├─ 各ルールの AICondition[] を AND 評価
│   ├─ probability による確率ゲーティング
│   ├─ マッチした場合: targetSelects[resultIndex] でターゲット選択
│   └─ マッチなし: 現ターゲット維持
│
├─ 3. 行動切替判定 (_actionJudgeTimer <= 0 の場合)
│   ├─ currentMode.actionRules[] を優先度順に評価
│   ├─ 各ルールの AICondition[] を AND 評価（ターゲット確定済み）
│   ├─ マッチした場合: actions[resultIndex] を実行
│   └─ マッチなし: actions[defaultActionIndex] を実行
│
└─ 4. 結果適用
    └─ controller.ProcessAIResult(resultData)
```

**設計根拠 — ターゲット→行動順序の理由（旧設計から変更）:**

旧設計では「行動→ターゲット」順だったが、新設計では **「ターゲット→行動」順** に変更した。

| 順序 | メリット |
|---|---|
| ターゲット確定後に行動選択 | 行動条件で「ターゲットとの距離」「ターゲットのHP」等の相対条件を使える |
| デフォルト行動の保証 | 条件全不一致でもデフォルト行動が存在 → 棒立ちしない |
| DamageScore はターゲット判断材料 | ヘイト代替の累積ダメージスコアはターゲット切替条件として自然に配置 |

> **判定システムの詳細**: [07_AI判定システム再設計](./07_AI判定システム再設計.md) 参照

### 3.5 条件システム（AIConditionType — 統一条件enum）

旧設計の3種の条件enum（ActTriggerCondition / TargetSelectCondition / MoveSelectCondition, 計41バリアント）を
12種の `AIConditionType` に統一。TargetFilter との組み合わせで同等以上の表現力を実現。

| AIConditionType | 説明 |
|---|---|
| `Count` | フィルター該当キャラ数 |
| `HpRatio` | HP割合（%） |
| `MpRatio` | MP割合（%） |
| `StaminaRatio` | スタミナ割合（%） |
| `ArmorRatio` | アーマー割合（%） |
| `Distance` | ターゲットとの距離 |
| `NearbyFaction` | 周囲の特定陣営キャラ数 |
| `ProjectileNear` | 飛び道具が一定距離内 |
| `ObjectNearby` | 指定オブジェクト/地形が周囲にあるか |
| `DamageScore` | 累積ダメージスコア（ヘイト代替） |
| `EventFired` | イベントフラグ |
| `SelfActState` | 自分の行動状態 |

### 3.6 ターゲット選択（AITargetSelect — ソートキー方式）

旧設計の TargetSelectCondition（28種）を、TargetSortKey + 属性指定 + 反転フラグで代替。

| TargetSortKey | 説明 |
|---|---|
| `Distance` | 距離順 |
| `HpRatio` / `HpValue` | HP割合 / HP実値順 |
| `AttackPower` / `DefensePower` | 攻撃力 / 防御力順（Element指定可） |
| `TargetingCount` | 狙われ数順 |
| `LastAttacker` | 最後に自分を攻撃した相手 |
| `DamageScore` | 累積ダメージスコア最大の相手 |
| `Self` / `Player` / `Sister` | 固定ターゲット |

### 3.9 TargetFilter（ターゲットフィルター）

参考コードの TargetFilter 設計を洗練して導入する。

```csharp
/// <summary>ターゲットフィルター条件。複数条件を AND/OR で組み合わせ可能。</summary>

{
```

### 3.11 PersonalityData との連携

PersonalityData の各パラメータが AI 判定に影響する箇所と補正式を定義する。

```
PersonalityData がAI判定に影響する箇所:

1. トリガー判定:
   aggression         → 戦闘モードへの遷移閾値を補正
   survivalPriority   → 撤退モードのHP閾値を補正

2. 行動選択:
   aggression         → 攻撃系行動の重みを補正
   resourceSaving     → スキル使用の重みを減算
   riskTolerance      → FeintCancel の重みを加算
   comboCommitment    → コンボ継続時のキャンセル確率を減算
   distancePreference → 移動行動の距離パラメータを補正

3. ターゲット選択（行動選択後、ActData.targetJudge に基づき実行）:
   revengeTendency    → 攻撃者へのヘイト加算倍率
   allyProtection     → 味方HP低下時のターゲット切り替え確率

4. フレームアドバンテージ判断:
   interruptPriority  → 有利フレーム時の割り込み攻撃確率

補正式:
  adjustedWeight = baseWeight × (1.0 + personality.relevantParam × 0.5 - 0.25)

  ※ personality 値が 0.5（中央）の場合、補正なし（× 1.0）
  ※ 0.0 の場合 ×0.75、1.0 の場合 ×1.25
  ※ baseWeight は ActData.weight で定義された基本重み
```

### 3.12 乱数生成

軽量かつ再現性のある乱数を生成するため、XorShift ベースのシード生成を使用する。

```csharp
uint seed = SeedGenerate((uint)(characterID + objectHash + (int)(nowTime * 1000)));
```

`actRatio`（実行確率 0〜100）と乱数を比較して確率ゲーティングを行う。100 の場合は必ず実行される。

## 4. モードシステム (CharacterModeData)

### 4.1 AIMode の構造（旧 CharacterModeData を置換）

各キャラクターは複数の **モード** を持ち、モードごとに異なる判定ルールセットが定義される。

```csharp
[Serializable]
public struct AIMode
{
    public string modeName;                // モード名（エディタ表示用）
    public AIRule[] targetRules;           // ターゲット切替ルール（優先度順）
    public AIRule[] actionRules;           // 行動切替ルール（優先度順）
    public AITargetSelect[] targetSelects; // ターゲット選択方法（targetRules から参照）
    public ActionData[] actions;           // 行動データ（actionRules から参照）
    public byte defaultActionIndex;        // デフォルト行動
    public float2 judgeInterval;           // x: ターゲット判定間隔, y: 行動判定間隔
}
```

BrainStatus（ScriptableObject）は `AIMode[]` 配列を保持し、キャラクター種別ごとに全モードの設定を定義する。

```csharp
// BrainStatus 内
public AIMode[] modes;
```

### 4.2 モード切替

モード切替は `ActionData.stateChange` が `ActState.モード変更` の場合に発動する。
行動ルールの結果としてモード変更が実行され、`ActionData.param` で指定されたモード番号に切り替わる。

```
行動ルールマッチ
  → actions[resultIndex] を取得
    → stateChange == モード変更: nowMode = action.param
    → stateChange == 攻撃/防御/移動等: 通常行動実行
```

### 4.3 モード別の判定ルール

各モードは独立した AIRule 配列を持つ。モードが切り替わると、ターゲットルール・行動ルールの両方が切り替わる。

| データ | アクセス方法 |
|---|---|
| 判定間隔 | `modes[nowMode].judgeInterval` |
| ターゲットルール | `modes[nowMode].targetRules[]` |
| 行動ルール | `modes[nowMode].actionRules[]` |
| デフォルト行動 | `modes[nowMode].actions[defaultActionIndex]` |

> **AIRule / AICondition / AITargetSelect の詳細定義**: [07_AI判定システム再設計](./07_AI判定システム再設計.md) セクション3 参照

### 4.4 BrainData のフラット化

旧設計と同様に、`AIMode[]`（モード数）のデータをフラット化してキャッシュ効率を維持する。
AIRule / AICondition は固定サイズ構造体のため、フラット化が容易である。

### 4.5 モード設定の例

| モード番号 | 名称 | 用途例 |
|---|---|---|
| 0 | 通常モード | 非戦闘時の巡回・待機 |
| 1 | 戦闘モード | 敵を検知した際の攻撃行動 |
| 2 | 逃走モード | HP 低下時に回復ポイントへ移動 |
| 3 | 警戒モード | 周囲に敵の気配がある場合の待機 |

## 5. 仲間キャラクターAI（カスタマイズ可能）

### 5.1 仲間AI の特殊要件

仲間キャラクター（プレイヤー陣営の NPC）は、通常の敵 AI とは異なる特殊な行動パターンが必要である。

| 要件 | 説明 |
|---|---|
| **追従（フォロー）** | プレイヤーとの距離が一定以上離れた場合、プレイヤー方向に移動する |
| **支援行動（回復・バフ）** | プレイヤーや味方の HP が閾値を下回った場合に回復行動を優先する |
| **連携攻撃** | プレイヤーが攻撃中のターゲットに対して追撃する。プレイヤーのターゲット情報を参照する |
| **攻撃積極性の設定** | プレイヤーが設定可能な積極性レベルに応じて、攻撃頻度と慎重さが変化する |
| **プレイヤー近接意識** | プレイヤーとの距離を常に監視し、離れすぎないように行動を調整する |

### 5.2 CompanionBehaviorSetting

仲間 AI 専用の行動設定パラメータを構造体として定義する。これは既存の `CharacterModeData` を拡張する形で使用される。

```csharp
/// <summary>
/// 仲間キャラクターの行動設定。
/// プレイヤーがカスタマイズ可能なパラメータを含む。
/// </summary>
public struct CompanionBehaviorSetting
{
    /// <summary>
    /// プレイヤーとの追従距離。
    /// この距離を超えるとプレイヤー方向に移動を開始する。
    /// </summary>
    public float followDistance;

    /// <summary>
    /// 支援行動（回復・バフ）の発動距離。
    /// 味方がこの範囲内にいる場合のみ支援行動を試みる。
    /// </summary>
    public float supportRange;

    /// <summary>
    /// 攻撃積極性レベル。
    /// 0 = 完全に防御的（攻撃しない）
    /// 50 = バランス型
    /// 100 = 完全に攻撃的（積極的に敵を攻撃）
    /// </summary>
    public byte aggressionLevel;

    /// <summary>
    /// 回復行動を発動する HP 閾値（パーセント）。
    /// 味方の HP がこの値以下になると回復を優先する。
    /// </summary>
    public byte healThreshold;

    /// <summary>
    /// プレイヤーが攻撃中のターゲットを優先するかどうか。
    /// true の場合、プレイヤーのターゲットに対する TargetSelectCondition の
    /// 優先度が上昇する。
    /// </summary>
    public BitableBool prioritizePlayerTarget;
}
```

### 5.3 仲間AIのモード定義

仲間 AI は以下のモードを持ち、トリガー条件によって自動的に切り替わる。

#### モード 0: 追従モード

プレイヤーに付いていく基本モード。

| 項目 | 設定 |
|---|---|
| トリガー条件 | プレイヤーとの距離が `followDistance` を超過 → 移動開始 |
| ターゲット | プレイヤー（TargetSelectCondition.プレイヤー） |
| 行動 | 移動（ActState.移動） |
| モード遷移 | 敵を検知 → 戦闘モードへ / 味方 HP 低下 → 支援モードへ |

#### モード 1: 戦闘モード

敵を攻撃するモード。`aggressionLevel` によって行動パターンが変化する。

| 項目 | 設定 |
|---|---|
| トリガー条件 | 敵キャラクター検知（RecognizeObjectType による認識） |
| ターゲット | `prioritizePlayerTarget` が true ならプレイヤーのターゲットを優先。それ以外はヘイト値順 |
| 行動 | `aggressionLevel` に応じて攻撃頻度を調整。低い場合は防御行動を挟む |
| モード遷移 | 敵全滅 → 追従モードへ / 自HP低下 → 防御モードへ / 味方HP低下 → 支援モードへ |

#### モード 2: 支援モード

回復・バフを優先するモード。

| 項目 | 設定 |
|---|---|
| トリガー条件 | 味方の HP が `healThreshold` 以下 |
| ターゲット | HP が最も低い味方（TargetSelectCondition.HP） |
| 行動 | 回復行動（ActState.支援）を優先。`supportRange` 内の味方のみ対象 |
| モード遷移 | 全味方 HP が閾値以上に回復 → 前モードに復帰 |

#### モード 3: 防御モード

プレイヤーを守るモード。

| 項目 | 設定 |
|---|---|
| トリガー条件 | 自HP低下、またはプレイヤーが多数の敵に囲まれている |
| ターゲット | プレイヤーを狙っている敵（対象のキャラの周囲に同陣営が一定以上密集している時） |
| 行動 | 防御行動（ActState.防御）とプレイヤー付近への移動を交互に実行 |
| モード遷移 | 状況安定 → 戦闘モードまたは追従モードへ |

### 5.4 仲間AIの判定フロー

```
┌─ 仲間AI AIBrain.Evaluate(nowTime) ───────────────────────────┐
│                                                               │
│  1. イベントによるモード変更チェック                          │
│     └─ eventContainer に modeOverride 付きイベントがあれば     │
│        nowMode を強制変更（トリガー判定より優先）              │
│                                                               │
│  2. プレイヤー位置・状態の参照                                │
│     ├─ GameManager.Data からプレイヤーの HP/位置を取得        │
│     └─ GameManager.Data からプレイヤー周囲の状況を取得        │
│                                                               │
│  3. CompanionBehaviorSetting の適用                           │
│     ├─ followDistance による距離チェック                      │
│     ├─ healThreshold による支援判定                          │
│     └─ aggressionLevel による攻撃/防御バランス調整            │
│                                                               │
│  4. 通常のモード判定（トリガー → 行動 → ターゲット）          │
│     ├─ トリガー条件にプレイヤー距離・味方HP条件を追加         │

│                                                               │

```

### 5.5 AIカスタマイズUI（設計）

プレイヤーが仲間AIの行動をカスタマイズするための UI 設計。

| パラメータ | UI 表示 | 入力形式 | 範囲 |
|---|---|---|---|
| `aggressionLevel` | 「攻撃積極性」 | スライダー | 0（防御的）〜 100（攻撃的） |
| `healThreshold` | 「回復タイミング」 | スライダー | 10% 〜 80% |
| `followDistance` | 「追従距離」 | スライダー | 近い（2.0）〜 遠い（10.0） |
| `prioritizePlayerTarget` | 「ターゲット連携」 | トグル | ON / OFF |
| `supportRange` | 「支援範囲」 | スライダー | 狭い（3.0）〜 広い（12.0） |

UI は **プリセット** も用意する。

| プリセット名 | aggressionLevel | healThreshold | followDistance | prioritizePlayerTarget |
|---|---|---|---|---|
| 「攻撃重視」 | 80 | 25 | 8.0 | true |
| 「バランス」 | 50 | 40 | 5.0 | true |
| 「回復重視」 | 20 | 60 | 4.0 | false |
| 「防御重視」 | 30 | 50 | 3.0 | false |

## 6. SensorSystem（認識システム）

### 6.1 概要

`SensorSystem` は各キャラクターの MonoBehaviour にアタッチされ、周囲の環境情報を収集して `RecognitionData` に反映するコンポーネントである。アビリティではなく、AI 判定のための入力データを生成する専用システムである。

```csharp
public class SensorSystem : MonoBehaviour
{
    [SerializeField] TriggerSensor2D _sightSensor;      // 視覚センサー
    [SerializeField] RangeSensor2D _environmentSensor;   // 環境センサー
    [SerializeField] float _closeRangeLimit;             // 近距離判定閾値
    [SerializeField] float _pulseInterval;               // スキャン間隔
    [SerializeField] byte _detectionLimit;               // 検知数上限
}
```

### 6.2 視覚センサー (TriggerSensor2D)

| 項目 | 説明 |
|---|---|
| 用途 | 戦闘時に攻撃や飛び道具を検知する |
| 動作 | トリガーコライダーによるリアルタイム検知 |
| 戦闘/非戦闘切替 | `ModeChange(bool isCombat)` でフィルターレイヤーを変更 |
| 戦闘時検知対象 | 攻撃体、飛び道具 |
| 非戦闘時検知対象 | キャラクター、飛び道具 |

### 6.3 環境センサー (RangeSensor2D)

| 項目 | 説明 |
|---|---|
| 用途 | 周囲の環境状況（キャラクター、地形、アイテム）を定期的にスキャンする |
| 動作 | `_pulseInterval` 間隔で `Pulse()` を発行し、距離順で検知結果を取得 |
| 近距離判定 | `_closeRangeLimit` の二乗距離で近距離キャラクターをカウント |
| 検知数制限 | `_detectionLimit` で検知するオブジェクト数の上限を設定 |

### 6.4 SensorAct() フロー

```csharp
public void SensorAct(ref RecognitionData recognition, float nowTime)
{
    // 1. 実行間隔チェック
    if (nowTime - _lastJudgeTime < _pulseInterval) return;

    // 2. 認識データをリセット
    recognition.Reset();

    // 3. 環境センサーを発火
    _environmentSensor.Pulse();

    // 4. 検出結果を距離順で取得
    Span<GameObject> detectObjects = _environmentSensor
        .GetDetectionsByDistance(_detectBuffer).AsSpan();

    // 5. 検出オブジェクトの分析
    DetectDataAnalyze(ref recognition, detectObjects, myPosition);

    // 6. 最終スキャン時間を更新
    _lastJudgeTime = nowTime;
}
```

### 6.5 RecognitionData 出力

`RecognitionData` は SoA コンテナに格納され、AIBrain から参照される。

## 7. イベントシステム

### 7.1 BrainEventFlagType

キャラクター間で発生するイベントをビットフラグで管理する。

```csharp
[Flags]
public enum BrainEventFlagType : byte
{
    None              = 0,
    大ダメージを与えた = 1 << 0,   // 相手に大きなダメージを与えた
    大ダメージを受けた = 1 << 1,   // 相手から大きなダメージを受けた
    キャラを倒した     = 1 << 2,   // 敵または味方を倒した
    回復を使用         = 1 << 3,   // 回復アビリティを使用した
    支援を使用         = 1 << 4,   // 支援アビリティを使用した
    攻撃対象指示       = 1 << 5,   // 指示による攻撃対象の指定
}
```

### 7.2 イベントの発行と消費

イベントは `BrainEventContainer` で管理される。

```csharp
public struct BrainEventContainer
{
    public BrainEventFlagType eventType;  // イベント種別
    public int targetHash;                // 対象キャラのハッシュ
    public float startTime;               // イベント開始時刻
    public float eventHoldTime;           // イベント保持期間
    public byte modeOverride;             // モード変更指定（0xFF = 指定なし）
}
```

`modeOverride` が `0xFF` 以外の場合、対象キャラの現在モードを強制的に切り替える。仲間AIのモード番号（0:追従, 1:戦闘, 2:支援, 3:防御）を指定できる。敵AIのモード切替にも使用可能。

**モード変更イベントの発行例:**
```csharp
// ボス戦突入時 → 仲間全員を戦闘モードへ
AIManager.Instance.RaiseBrainEvent(
    companionHash,
    BrainEventFlagType.None,
    holdTime: 0f,
    modeOverride: 1  // 戦闘モード
);

// 回復ポイント到達時 → 仲間を支援モードへ
AIManager.Instance.RaiseBrainEvent(
    companionHash,
    BrainEventFlagType.None,
    holdTime: 0f,
    modeOverride: 2  // 支援モード
);
```

**イベントライフサイクル:**

```
1. 発行: イベント発生時に eventContainer に追加
   → BrainEventContainer(eventType, hashCode, holdTime, modeOverride)

2. モード変更: modeOverride != 0xFF なら対象キャラの nowMode を即時変更
   → トリガー判定より先に処理される（最優先）
   → 適用後は modeOverride を 0xFF にリセット

3. 参照: AIBrain 内のトリガー判定で ActTriggerCondition.一定のイベントが発生した時 を評価
   → 対象キャラの CharacterStateInfo.brainEvent フラグを確認

4. 消費: 時間経過（nowTime - startTime >= eventHoldTime）でイベントを削除
   → または対象キャラの死亡時に CharacterDead() 内で削除

5. 削除: 後方からのループで RemoveAtSwapBack で安全に削除
```

### 7.3 ヘイト管理

ヘイトシステムは2層構造で管理される。

| 種別 | キー | 説明 |
|---|---|---|
| **チームヘイト** (`teamHate`) | `int2(チーム番号, 対象hash)` | チーム全体で共有するヘイト値。敵対チーム登録時に初期値10で作成 |
| **個人ヘイト** (`personalHate`) | `int2(自分hash, 相手hash)` | 個人間のヘイト値。ダメージ量に比例して増減 |

**ヘイトの増減条件:**

- ダメージを受けた → 攻撃者へのヘイト増加
- ダメージを与えた → RecognitionData.hateEnemyHash に反映（ヘイト上昇側は 1.2 倍、下降側は 0.8 倍のスコアで評価）
- キャラ死亡 → 該当キャラへのヘイトエントリを全チームから削除
- BrainEventFlagType の `大ダメージを与えた` / `大ダメージを受けた` でヘイト変動のイベントをトリガーに反映

**敵対チェック:**

```csharp
private bool CheckTeamHostility(int team1, int team2)
{
    return (relationMap[team1] & (1 << team2)) > 0;
}
```

`relationMap` はチームごとに1エントリの `int[]` で、ビット位置がチーム番号に対応する。

## 9. パフォーマンス最適化

### 9.1 SoA キャッシュ効率

| 手法 | 効果 |
|---|---|
| **Structure of Arrays** | 同種のデータが連続配置されるため、キャッシュラインの利用効率が向上 |
| **BrainData のフラット化** | ジャグ配列 → 単一配列に変換。スライスアクセスで参照 |
| **SequentialLayout** | `[StructLayout(LayoutKind.Sequential)]` で構造体のメモリレイアウトを保証 |
| **判定タイマーの SoA 外配置** | 判定クールダウン（旧 CharacterColdLog）は `JudgmentLoop` ローカル float で十分、SoA 化しない |

### 9.2 判定間隔制御

全ての判定を毎フレーム実行するのではなく、`JudgmentLoop` のローカルタイマーで制御する。

```csharp
// JudgmentLoop のフィールド（ピュアロジック、SoA 外）
private float _targetJudgeTimer;
private float _actionJudgeTimer;

public void Tick(float deltaTime, float nowTime)
{
    _targetJudgeTimer -= deltaTime;
    _actionJudgeTimer -= deltaTime;

    AIMode currentMode = /* 現在モード */;

    if (_targetJudgeTimer <= 0f)
    {
        _targetJudgeTimer = currentMode.judgeInterval.x;
        /* ターゲット切替判定 */
    }
    if (_actionJudgeTimer <= 0f)
    {
        _actionJudgeTimer = currentMode.judgeInterval.y;
        /* 行動切替判定 */
    }
}
```

これにより、キャラクター数が多い場合でもフレームあたりの計算量を分散できる。

### 10.2 アクションマスキング

#### 10.2.1 概要

`AnimationStateData.currentPhase` に基づいて、物理的に不可能な行動をマスクする。
ルールベース AI が物理的に不可能な行動を選択しないように制御する。

#### 10.2.2 マスクテーブル

| AnimationPhase | Attack | Move | Guard | Dodge | Heal | Buff | Wait | Flee | FeintCancel |
|---|---|---|---|---|---|---|---|---|---|
| **Neutral** | OK | OK | OK | OK | OK | OK | OK | OK | -- |
| **Anticipation (cancelable)** | -- | -- | -- | -- | -- | -- | -- | -- | OK |
| **Anticipation (committed)** | -- | -- | -- | -- | -- | -- | -- | -- | -- |
| **Active** | -- | -- | -- | -- | -- | -- | -- | -- | -- |
| **Recovery** | -- | -- | OK | OK | -- | -- | OK | OK | -- |

- `OK` = 選択可能、`--` = マスク済み（選択不可）
- `Anticipation (cancelable)` は `isCancelable == true` の場合のみ FeintCancel が有効
- `Anticipation (committed)` と `Active` は全行動がマスクされる（現在の行動が完了するまで待機）

#### 10.2.3 Evaluate() 内のマスキング処理

```csharp
/// <summary>
/// AnimationPhase に基づくアクションマスクの適用。
/// Evaluate() の行動判定フェーズ冒頭で呼び出す。
/// </summary>
private void ApplyActionMask(
    AnimationStateData animState,
    Span<bool> actionMask,
    int actionCount)
{
    // デフォルト: 全行動を有効化
    for (int i = 0; i < actionCount; i++)
        actionMask[i] = true;

    switch (animState.currentPhase)
    {
        case AnimationPhase.Neutral:
            // 全行動が選択可能。FeintCancel のみ無効
            // （Neutral 中にキャンセルする対象がない）
            actionMask[(int)ActType.FeintCancel] = false;
            break;

        case AnimationPhase.Anticipation:
            // 全行動を無効化
            for (int i = 0; i < actionCount; i++)
                actionMask[i] = false;

            // キャンセル可能な場合のみ FeintCancel を許可
            if (animState.isCancelable)
                actionMask[(int)ActType.FeintCancel] = true;
            break;

        case AnimationPhase.Active:
            // 攻撃判定中: 全行動を無効化（行動完了を待つ）
            for (int i = 0; i < actionCount; i++)
                actionMask[i] = false;
            break;

        case AnimationPhase.Recovery:
            // 硬直中: 防御系行動のみ許可
            for (int i = 0; i < actionCount; i++)
                actionMask[i] = false;

            actionMask[(int)ActType.Guard] = true;
            actionMask[(int)ActType.Dodge] = true;
            actionMask[(int)ActType.Wait]  = true;
            actionMask[(int)ActType.Flee]  = true;
            break;
    }
}
```

#### 10.2.4 行動判定への統合

```
Evaluate(nowTime)
│
├─ ... （トリガー判定は既存のまま）
│
├─ 3. 行動判定 (passTime.x >= judgeIntervals.x の場合)
```

### 10.3 遅延バッファ処理

#### 10.3.1 概要

`DeliberationBuffer` は AIBrain の出力（`MovementInfo`）を即座に実行せず、
人間的な反応遅延を挿入する。難易度設定に応じた基本遅延とランダム幅で遅延量を決定する。

#### 10.3.2 処理フロー

```
AIBrain.Evaluate() 完了
  │
  ▼
┌─ ProcessDeliberationBuffers() ──────────────────────────────────────┐
│                                                                      │
│  各キャラクター i について:                                          │
│                                                                      │
│  ┌─ 新規行動の格納 ─────────────────────────────────────────────┐   │
│  │ if (judgeResult[i] が有効な新規行動)                         │   │
│  │ {                                                            │   │
│  │   buffer.pendingAction = judgeResult[i];                     │   │
│  │   baseDelay = buffer.baseDelayFrames;                        │   │
│  │   variance  = Random(0, buffer.varianceFrames);              │   │
│  │   buffer.currentDelay = (byte)(baseDelay + variance);        │   │
│  │   buffer.hasPendingAction = true;                            │   │
│  │ }                                                            │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  ┌─ 遅延カウントダウン ──────────────────────────────────────────┐  │
│  │ if (buffer.hasPendingAction && buffer.currentDelay > 0)       │  │
│  │     buffer.currentDelay--;                                    │  │
│  │                                                               │  │
│  │ if (buffer.hasPendingAction && buffer.currentDelay == 0)      │  │
│  │ {                                                             │  │
│  │     BaseCharacter[i].ProcessAIResult(buffer.pendingAction);   │  │
│  │     buffer.hasPendingAction = false;                          │  │
│  │ }                                                             │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

#### 10.3.3 実装コード

```csharp
/// <summary>
/// 遅延バッファの処理。AIBrain.Evaluate() 完了後、メインスレッドで実行する。
/// 毎フレーム呼び出し、遅延カウントダウンと行動実行を管理する。
/// </summary>
private void ProcessDeliberationBuffers()
{
    int count = this.characterDataDictionary.Count;

    for (int i = 0; i < count; i++)
    {
        DeliberationBuffer buffer = deliberationBufferList[i];
        MovementInfo result = judgeResult[i];

        // ─── 新規行動の格納 ───
        // judgeResult に有効な行動が出力されている場合
        if (result.actNum >= 0)
        {
            buffer.pendingAction = result;

            // 遅延フレーム計算: 基本遅延 + ランダム幅
            uint seed = SeedGenerate((uint)(i + (int)(nowTime * 1000)));
            int variance = (int)(seed % (uint)(buffer.varianceFrames + 1));
            int totalDelay = buffer.baseDelayFrames + variance;
            totalDelay = math.clamp(totalDelay, 0, 255);

            buffer.currentDelay = (byte)totalDelay;
            buffer.hasPendingAction = true;
        }

        // ─── 遅延カウントダウン ───
        if (buffer.hasPendingAction)
        {
            if (buffer.currentDelay > 0)
            {
                buffer.currentDelay--;
            }
            else
            {
                // 遅延完了 → 行動実行
                characterControllers[i].ProcessAIResult(buffer.pendingAction);
                buffer.hasPendingAction = false;
            }
        }

        deliberationBufferList[i] = buffer;
    }
}
```

#### 10.3.4 難易度別の遅延設定例

| 難易度 | baseDelayFrames | varianceFrames | 実効遅延（60fps） |
|---|---|---|---|
| Easy | 12 | 6 | 200〜300ms |
| Normal | 6 | 4 | 100〜167ms |
| Hard | 2 | 2 | 33〜67ms |
| Expert | 0 | 1 | 0〜17ms |

### 10.4 Blender 改良（条件付きパラメータ統合）

#### 10.4.1 従来の Blender

v7 以前の Blender は条件配列（`low_hp`, `high_resource` 等）から最終的な 10D パラメータを直接生成していた。

```
従来: 条件配列 → Blender → 最終 10D パラメータ → AIBrain 入力
```

#### 10.4.2 改良後の Blender（ベース + デルタ方式）

改良後は Blender が **デルタのみ** を生成する。ベース値は LLM 由来の `PersonalityData` である。

```
改良後:
  PersonalityData（LLMベース）   ← LLM が生成したキャラクター性格
+ Blender delta（条件ベース）     ← 状況に応じた微調整（low_hp → survivalPriority +0.2 等）
= 最終 PersonalityData            → AIBrain 入力
```

#### 10.4.3 デルタ合成の計算コード

```csharp
/// <summary>
/// 2つのソースからの PersonalityData を合成する。
/// Blender はデルタ（差分）のみを生成し、
/// LLM ベースに加算してクランプする。
/// </summary>
private PersonalityData ComposeFinalPersonality(
    PersonalityData llmBase,       // LLM 出力（不変のベース値）
    PersonalityData blenderDelta)  // 条件配列からの微調整
{
    PersonalityData result = llmBase;

    // ─── Blender デルタの加算 ───
    result.aggression        = ClampHalf(result.aggression        + blenderDelta.aggression);
    result.survivalPriority  = ClampHalf(result.survivalPriority  + blenderDelta.survivalPriority);
    result.distancePreference= ClampHalf(result.distancePreference+ blenderDelta.distancePreference);
    result.resourceSaving    = ClampHalf(result.resourceSaving    + blenderDelta.resourceSaving);
    result.revengeTendency   = ClampHalf(result.revengeTendency   + blenderDelta.revengeTendency);
    result.interruptPriority = ClampHalf(result.interruptPriority + blenderDelta.interruptPriority);
    result.allyProtection    = ClampHalf(result.allyProtection    + blenderDelta.allyProtection);
    result.resourceUnlock    = ClampHalf(result.resourceUnlock    + blenderDelta.resourceUnlock);
    result.comboCommitment   = ClampHalf(result.comboCommitment   + blenderDelta.comboCommitment);
    result.riskTolerance     = ClampHalf(result.riskTolerance     + blenderDelta.riskTolerance);

    return result;
}

/// <summary>half 値を [0, 1] にクランプするユーティリティ</summary>
private static half ClampHalf(half value)
{
    return (half)math.clamp((float)value, 0f, 1f);
}
```

## 11. State Machine 選択指針（一般論との対比）

> Sources: nice-wolf-studio/unity-claude-skills (MIT) — unity-state-machines を SisterGame の SoA + Ability 拡張前提で再構成

SisterGame の AI システムは **MonoBehaviour ベースのルール駆動 AI**（前述）を採用しているため、汎用的な FSM/HFSM/BT パターンとは異なる設計思想をとる。
ただし**サブシステム（敵単体の細かい挙動・サブ行動・コンボ判定など）**で局所的に状態機械を導入する場面はあり得るため、選択指針を明記する。

### 11.1 サイズ別ガイド（一般論）

| 状態数 | 推奨実装 | SisterGame 内での適用例 |
|---|---|---|
| 1-3 | bool フラグ / single enum | `_isJumping`、`_isAttacking` 単体管理 |
| 3-5 | enum + switch | コンボステップ管理（攻撃 1 段→2 段→3 段）|
| 5-15 | Interface FSM (`IState` パターン) | 敵単体の挙動（巡回 / 警戒 / 追跡 / 攻撃 / 撤退）|
| 15+ / 階層的 | HFSM（Hierarchical FSM）| ボス戦のフェーズ + フェーズ内ステート |
| 優先度ベース | Behavior Tree | 複雑な意思決定（複数ターゲット、多条件分岐）|
| UI / ポーズ | Stack-based | 設定 → グラフィック → 解像度 のサブメニュー進行 |

### 11.2 SisterGame の方針

| 場面 | 採用 | 理由 |
|---|---|---|
| **キャラクター AI 判定（メイン）** | ❌ 汎用 FSM / BT を使わない | 既存の `AIBrain.Evaluate()` + ルール駆動 + DeliberationBuffer がメインスレッドで動く設計のため。FSM ノードを別途持つと SoA + objectHash アクセスの効率が崩れる |
| **コンボ / 攻撃ステート管理** | ⭕ enum + switch | 状態数が少なく（1-5 段階）、Ability 拡張で十分管理可能 |
| **モード切替（Companion AI）** | ⭕ ScriptableObject ベース（既存 `CharacterModeData`）| ルール駆動と整合、ScriptableObject で Designer が編集可能 |
| **UI / メニュー画面遷移** | ⭕ Stack-based pattern（`MenuStackManager`）| 戻るボタン処理が自然に実装可能、既存実装あり |
| **ボス挙動（フェーズ移行）** | ⭕ HFSM 検討余地あり | 大型ボス追加時に独立したサブシステムとして導入可（SoA AI とは別レイヤー）|

### 11.3 実装規約

#### 「plain C# classes, NOT MonoBehaviours」原則

State クラスは **MonoBehaviour ではなく素の C# クラス**として実装する。

```csharp
// OK: 素の C# クラスとして実装
public interface IEnemyState
{
    void Enter(EnemyContext context);
    void Update(EnemyContext context, float dt);
    void Exit(EnemyContext context);
}

public class PatrolState : IEnemyState
{
    public void Enter(EnemyContext ctx) { /* ... */ }
    public void Update(EnemyContext ctx, float dt) { /* ... */ }
    public void Exit(EnemyContext ctx) { /* ... */ }
}

// 呼出側は Ability 拡張から（MonoBehaviour ではない）
```

理由:
- SoA データへのアクセスは `GameManager.Data` 経由（State クラスがコンテキストを保持しない）
- MonoBehaviour にすると inspector ノイズ / Update 重複 / ライフサイクル管理が必要になる
- 単体テスト（Edit Mode）が容易になる
- Ability 拡張パターンと整合（State も「能力としてキャラクターに注入」できる）

#### 外部ライブラリ非依存

SisterGame は **NodeCanvas / Behavior Designer / FSM Graph 等の外部ライブラリを採用しない**:
- visual editor は管理コストが高い（メモリ feedback_animation-control-approach と同じ理由）
- C# コードでロジックを管理する方が AI / シナリオの組み合わせ自由度が高い
- 設計レベルのドキュメント（`Architect/`）と feature-db のトレーサビリティが取りやすい

ただし、State machine の**選択指針**（FSM / HFSM / BT のどれを使うか）の判断は本セクション参照。

# 09 - アニメーション規約

> Sources: nice-wolf-studio/unity-claude-skills (MIT) — unity-animation を SisterGame の AnimationBridge 駆動方式に合わせて再構成

## 1. SisterGame の Animation 規約（正典）

SisterGame は **Animator Controller のトランジション配線を使用しない**。

### 1.1 採用方式: AnimationBridge + CrossFade 駆動

**理由**（メモリ feedback_animation-control-approach.md, 2026-04-23 確定）:
- Animator のトランジション設定はビジュアルエディタでの管理コストが高い（条件式の可視化が悪い、マージコンフリクトの解析が困難）
- ロジック側で状態管理した方が AI / シナリオ / 戦闘システム の組み合わせ自由度が高い
- C# コードでアニメーション遷移を記述すると Edit Mode テストが書ける
- AnimationBridge を介して論理的な状態管理 + `Animator.CrossFade()` 駆動

### 1.2 採用 API（許可）

| API | 用途 |
|---|---|
| `Animator.CrossFade(stateName, transitionDuration)` | クロスフェード遷移（標準）|
| `Animator.CrossFadeInFixedTime(stateName, fixedTime)` | 固定秒数のフェード |
| `Animator.Play(stateName)` | 即時切替（フェードなし、特殊用途）|
| `Animator.SetFloat / SetInt / SetBool / SetTrigger` | パラメータ駆動（**ただし Transition の発火条件としては使わない**、BlendTree のブレンド入力として限定使用）|
| BlendTree（1D / 2D / Direct） | 移動方向 / 速度などの連続値ブレンド |
| Inverse Kinematics（IK） | 必要な場面のみ |
| Root Motion | 必要な場面のみ（ただし物理演算と競合に注意）|

### 1.3 不採用要素（禁止）

| 要素 | 理由 |
|---|---|
| AnimatorController の **Transitions** | ロジック側で状態管理する方針のため。配線管理コストが高い |
| Parameters 駆動の **Conditional Transitions** | 同上。`SetTrigger("Jump")` で遷移するパターンを使わない |
| **StateMachineBehaviour** (`OnStateEnter` / `OnStateExit` / `OnStateUpdate`) | ロジックを Animator State に紐付けると分散しすぎる。AnimationBridge でロジックを集約する方が保守性が高い |
| Animator Override Controller の動的差替え | キャラごとの Override は許可するが、ランタイムの動的差替えは複雑度が増すため避ける |

### 1.4 AnimationBridge の役割

```csharp
// 概念実装（実装詳細は Assets/MyAsset/Runtime/Animation/AnimationBridge.cs を参照）
public class AnimationBridge
{
    private readonly Animator _animator;

    public AnimationBridge(Animator animator) { _animator = animator; }

    public void PlayIdle()    => _animator.CrossFade("Idle", 0.1f);
    public void PlayRun()     => _animator.CrossFade("Run",  0.1f);
    public void PlayAttack(int comboStep)
    {
        string stateName = comboStep switch
        {
            1 => "Attack1",
            2 => "Attack2",
            3 => "Attack3",
            _ => "Idle"
        };
        _animator.CrossFade(stateName, 0.05f);
    }
    public void PlayHurt()    => _animator.Play("Hurt"); // 即時切替（hitstun）
}
```

### 1.5 BaseCharacter との統合

`02_ベースキャラクターとアクションシステム.md § 8` で定義された `AnimationStateData`（SoA への書き戻し）と AnimationBridge は補完関係:

- **AnimationBridge**: Animator への命令側（CrossFade を呼ぶ）
- **BaseCharacter § 8**: Animator の現在状態を SoA に書き戻す側（AI判定 / アクションマスキング用）

両方が必要なのは、命令と状態確認が**非同期**だから（CrossFade を呼んでもすぐに State Machine が遷移するわけではなく、補間中の状態がある）。

## 2. Mecanim 一般知識（参考）

> 以下は SisterGame で**実装する場面で参照する**Mecanim の API リファレンス。
> ただし「**1.2 採用 API**」の範囲内のみ使う。Transitions / Parameters 駆動は不採用。

### 2.1 BlendTree

連続値（移動速度・方向 など）でアニメーションをブレンドするために使用。SisterGame では以下の場面で採用:
- **1D BlendTree**: Idle ⇄ Walk ⇄ Run の速度ブレンド
- **2D BlendTree**: 8 方向移動など、方向ベクトルでのブレンド
- **Direct BlendTree**: パーツ別に独立したブレンド係数（顔・上半身・下半身など）

### 2.2 IK（Inverse Kinematics）

`OnAnimatorIK(int layerIndex)` フックで実装。SisterGame では:
- 武器を持つ手を IK で武器グリップ位置に固定（Two-Bone IK）
- 視線追従（Look IK）
- 必要に応じて足の地面適応（Foot IK、ただし 2D アクションでは限定使用）

### 2.3 Root Motion

アニメーションクリップ自体に移動を含める方式。SisterGame での扱い:
- **物理（Rigidbody2D + Collider）と競合する**ため、**標準は Root Motion OFF**
- 例外: Cinematic Cutscene / イベントシーン内でのキャラクター移動は Root Motion 可
- 戦闘中の移動は Ability システム（JumpAbility / DashAbility 等）が担当

### 2.4 Avatar Mask

`AvatarMask` で特定の bone のみアニメーションを上書き。SisterGame での用途:
- 移動中に上半身だけ攻撃モーション再生（Layer 重ね合わせ）
- Companion AI のリアクション（顔の向きだけ Player を追う）

## 3. AnimationStateData との連動（実装詳細）

`AnimationPhase` enum と `MotionInfo`（攻撃のタイムライン）の連動は `02_ベースキャラクターとアクションシステム.md § 8` を参照。

要約:
- 攻撃開始時に `MotionInfo` から preMotion / activeMotion / recovery の各時間を取得
- 毎フレーム残りフレーム数を SoA に書き戻す
- AI判定は SoA 経由で「対戦相手は今 recovery 中だから攻撃チャンス」等を判定

---

## 4. Timeline の扱い

イベントシーン / Cutscene のみ Timeline を使う。`/create-event` skill で生成される構成と整合。
Gameplay 中のキャラクターアニメーションには Timeline を使わない（AnimationBridge 駆動を貫く）。

## 5. テスト戦略

AnimationBridge は MonoBehaviour 非依存の薄い wrapper として設計するため、**Edit Mode テスト可能**:

```csharp
[Test]
public void AnimationBridge_PlayAttack_CallsCrossFadeWithCorrectState()
{
    var mockAnimator = new MockAnimator(); // テスト用 Fake
    var bridge = new AnimationBridge(mockAnimator);

    bridge.PlayAttack(2);

    Assert.AreEqual("Attack2", mockAnimator.LastCrossFadeState);
    Assert.AreEqual(0.05f, mockAnimator.LastCrossFadeDuration);
}
```

Animator State 遷移自体は Play Mode テスト or 手動確認となる（unity-mcp 経由の auto-input test も活用可）。

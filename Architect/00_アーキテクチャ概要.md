## 1. 設計方針

### 1.1 コア原則

| 原則 | 説明 |
|---|---|
| **SoA（Structure of Arrays）+ SourceGenerator** | キャラクターデータをフィールドごとの配列で管理し、キャッシュ効率を最大化する。SourceGeneratorにより情報クラスに属性を付けるだけでSoAコンテナ・アクセサが自動生成され、手書きボイラープレートを排除する |
| **GameManager シングルトン（中央ハブ）** | GameManagerが全マネージャーとデータコンテナへの唯一の参照点。すべてのゲームシステムはGameManager経由でアクセスする。散在するシングルトンを排除し、依存関係を一元管理する |
| **ハッシュベースデータアクセス（GetComponent排除）** | GameObjectのhashCodeをキーとしてGameManager.Data経由でO(1)データアクセスを実現する。GetComponentによる毎フレーム検索コストを完全に排除する |
| **コンポーネント拡張（Abilityシステム）** | ベースキャラクターは最小限の責務のみ持つ。ジャンプ・ダッシュ等はAbilityコンポーネントとして追加・組み合わせ可能。新しい行動の追加が既存コードに影響しない |
| **データ駆動（ScriptableObject）** | ゲームバランスに関わる値はすべてScriptableObject上の情報クラスで定義する。コード変更不要でデザイナーが調整可能 |
| **単一責任** | 各クラス・構造体は1つの責務のみを持つ。機能重複を徹底排除する |

### 1.2 コーギーエンジンとの決別

コーギーエンジンは汎用2Dプラットフォーマーフレームワークとして多機能だが、以下の問題がある:

- **GetComponent多用**: 毎フレームのGetComponentによるパフォーマンスコスト
- **密結合**: 各コンポーネントがCorgiControllerに強く依存
- **AI非対応**: 柔軟なルールベースAI処理に対応できないアーキテクチャ
- **拡張困難**: CharacterAbilityの継承ツリーが深く、カスタマイズが困難

本設計では以下のように自前設計で置き換える:

| コーギーエンジンの機能 | 本設計での対応 | 改善点 |
|---|---|---|
| CorgiController | BaseCharacter + Ability コンポーネント群 | GameManager経由のデータアクセス、GetComponent不要 |
| CharacterAbility (継承ベース) | IAbility インターフェース + AbilityBase 基底クラス | コンポジションベースで自由に組み合わせ可能 |
| Health | DamageReceiver + CharacterBaseInfo (SoAコンテナ内) | HP/MP/スタミナをSoA配列で管理 |
| AIBrain / AIDecision / AIAction | MonoBehaviour ベースの AIBrain + AIInfo（設定） | SoACharaDataDic経由でデータアクセス |
| Weapon / MeleeWeapon / ProjectileWeapon | WeaponHolder + IWeaponBehaviour + AttackInfo | 武器データをScriptableObjectで定義、動的切り替え |
| DamageOnTouch | DamageDealer + DamageReceiver | 多属性対応、パリィ・ガード統合 |
| InputManager | 別途 InputSystem 連携（本設計スコープ外） | - |

## 2. GameManager - 全体管理の中心

### 2.1 設計思想

GameManagerは本アーキテクチャの**唯一の中央アクセスポイント**である。

従来のUnity開発では各システムがそれぞれシングルトンを持ったり、FindObjectOfTypeやGetComponentで互いを検索したりしていた。本設計ではGameManagerが全マネージャーと全データコンテナへの参照を保持し、あらゆるコードからの統一的なアクセス経路を提供する。

```
[従来のUnity設計]
  各スクリプトが自力で参照を探す
  MonoA → GetComponent<Health>() → HP取得
  MonoB → FindObjectOfType<AIManager>() → AI参照取得
  MonoC → AIManager.Instance.Container.Get(...) → データ取得
  → 散在するシングルトン、FindやGetComponentの頻繁な呼び出し

[本設計]
  GameManagerが全参照を一元管理
  どのコードからでも → GameManager.Data.GetXxx(hashCode)
  → 単一のアクセス経路、O(1)ハッシュルックアップ
```

### 2.2 クラス定義

```csharp
/// <summary>
/// ゲーム全体を管理するシングルトン。
/// 全データコンテナへの唯一のアクセスポイント。
/// </summary>
public class GameManager : MonoBehaviour, IDisposable
{
    // ──────────────────────────────────────────
    // シングルトン
    // ──────────────────────────────────────────
    public static GameManager Instance { get; private set; }

    // ──────────────────────────────────────────
    // データコンテナへのショートカット
    // ──────────────────────────────────────────
    /// <summary>
    /// キャラクターデータコンテナ。
    /// 使用例: GameManager.Data.GetBaseInfo(objectHash)
    /// </summary>
    public static SoACharaDataDic Data => Instance._characterContainer;

    // ──────────────────────────────────────────
    // サブマネージャー
    // ──────────────────────────────────────────
    [SerializeField] private AIManager _aiManager;

    // ──────────────────────────────────────────
    // データコンテナ
    // ──────────────────────────────────────────
    private SoACharaDataDic _characterContainer;

    // ──────────────────────────────────────────
    // 初期化
    // ──────────────────────────────────────────
    private void Awake()
    {
        if (Instance != null) { Destroy(gameObject); return; }
        Instance = this;
        DontDestroyOnLoad(gameObject);

        // データコンテナ生成
        _characterContainer = new SoACharaDataDic(initialCapacity: 64);

        // サブマネージャー初期化（コンテナを渡す）
        _aiManager.Initialize(_characterContainer);
    }

    // ──────────────────────────────────────────
    // キャラクター登録
    // ──────────────────────────────────────────
    /// <summary>
    /// キャラクターを登録し、データコンテナ内のインデックスを返す。
    /// BaseCharacter.Initialize() から呼ばれる。
    /// </summary>
    public int RegisterCharacter(CharacterInfo info, AIInfo aiInfo,
                                  BaseCharacter character)
    {
        int hash = character.gameObject.GetHashCode();
        int index = _characterContainer.Add(hash, character, info);
        _aiManager.OnCharacterRegistered(hash, aiInfo);
        return index;
    }

    /// <summary>キャラクター削除</summary>
    public void UnregisterCharacter(int objectHash)
    {
        _characterContainer.Remove(objectHash);
        _aiManager.OnCharacterRemoved(objectHash);
    }

    // ──────────────────────────────────────────
    // マネージャーアクセス
    // ──────────────────────────────────────────
    public AIManager AI => _aiManager;

    // ──────────────────────────────────────────
    // 破棄
    // ──────────────────────────────────────────
    public void Dispose()
    {
        _characterContainer?.Dispose();
        _aiManager?.Dispose();
    }

    private void OnDestroy()
    {
        Dispose();
        if (Instance == this) Instance = null;
    }
}
```

### 2.3 データアクセスパターン

```
┌──────────────────────────────────────────────────────────────────┐
│  従来の Unity パターン (GetComponent 地獄)                        │
│                                                                  │
│  var hp = target.GetComponent<Health>().currentHp;     ← 検索   │
│  var atk = target.GetComponent<Attack>().power;        ← 検索   │
│  var def = target.GetComponent<Defense>().armor;       ← 検索   │
│  ※ 毎回 O(N) のコンポーネント検索が発生                          │
└──────────────────────────────────────────────────────────────────┘

                            ↓ 本設計 ↓

┌──────────────────────────────────────────────────────────────────┐
│  本設計のパターン (ハッシュ → GameManager.Data)                   │
│                                                                  │
│  int hash = target.GetHashCode();  // or 保持済みのhash          │
│  ref var hp  = ref GameManager.Data.GetBaseInfo(hash);  ← O(1)  │
│  ref var atk = ref GameManager.Data.GetAtkStatus(hash); ← O(1)  │
│  ref var def = ref GameManager.Data.GetDefStatus(hash); ← O(1)  │
│  ※ ハッシュテーブルで O(1) ルックアップ。ref返しで直接書き込み   │
└──────────────────────────────────────────────────────────────────┘
```

### 2.4 各コンポーネントからのアクセス例

```csharp
// ---- BaseCharacter 内 ----
public bool IsAlive => GameManager.Data.GetBaseInfo(ObjectHash).currentHp > 0;

// ---- WeaponHolder 内 ----
private int _ownerHash;  // Initialize時に保存
public int GetOwnerAttack()
    => GameManager.Data.GetAtkStatus(_ownerHash).dispAtk;

// ---- DamageReceiver 内 ----
private int _ownerHash;
public void TakeDamage(DamageData data)
{
    ref var baseInfo = ref GameManager.Data.GetBaseInfo(_ownerHash);
    ref var defStatus = ref GameManager.Data.GetDefStatus(_ownerHash);
    // ダメージ計算...
    baseInfo.currentHp -= finalDamage;
}

// ---- 外部コード（UIなど）----
// hashさえ知っていれば、対象キャラのデータにアクセス可能
public void UpdateHpBar(int targetHash)
{
    var info = GameManager.Data.GetBaseInfo(targetHash);
    _hpSlider.value = info.hpRatio / 100f;
}
```

## 3. データアクセスフロー

### 3.1 従来方式との比較

```
┌────────────────────────────────────────────────────────────────┐
│ 比較: データアクセスコスト                                      │
├──────────────────────────┬─────────────────────────────────────┤
│ 従来 (GetComponent)      │ 本設計 (Hash-based)                 │
├──────────────────────────┼─────────────────────────────────────┤
│ GetComponent<T>()        │ Dictionary<int,int> lookup           │
│  → 型検索 O(N)           │  → hashCode → index O(1)           │
│  → コンポーネントリスト走査│  → 配列直接アクセス array[index]    │
│  → GC Alloc の可能性     │  → GC Alloc なし                   │
│  → キャッシュミス多発     │  → SoA配列で連続メモリ             │
│  → 拡張困難              │  → hash/indexで直接アクセス可能     │
└──────────────────────────┴─────────────────────────────────────┘
```

### 3.2 登録フロー

```
┌─────────────────────────────────────────────────────────────────┐
│                    登録フロー（ゲーム開始時）                     │
│                                                                 │
│  CharacterInfo (ScriptableObject)                               │
│       │                                                         │
│  BaseCharacter.Initialize()                                     │
│       │ objectHash = gameObject.GetHashCode()                   │
│       │                                                         │
│       ▼                                                         │
│  GameManager.Instance.RegisterCharacter(info, aiInfo, this)     │
│       │                                                         │
│       ├─→ SoACharaDataDic.Add(hash, this, info)                │
│       │     ├─ CharacterBaseInfo を info から生成して格納        │
│       │     ├─ SolidData を info から生成して格納               │
│       │     ├─ MoveStatus を info から生成して格納              │
│       │     ├─ StaminaInfo を info から生成して格納             │
│       │     ├─ CharacterAtkStatus を格納                        │
│       │     ├─ CharacterDefStatus を格納                        │
│       │     └─ ... (全構造体を格納)                             │
│       │                                                         │
│       └─→ return index                                          │
│             │                                                   │
│       BaseCharacter.CharIndex = index   ← 以降このindexで直接  │

└─────────────────────────────────────────────────────────────────┘
```

### 3.3 実行時アクセスフロー

```
┌─────────────────────────────────────────────────────────────────┐
│                    実行時アクセスフロー                           │
│                                                                 │
│  任意のコード（hash を知っている）                               │
│       │                                                         │
│       ▼                                                         │
│  GameManager.Data.GetBaseInfo(hash)                              │
│       │                                                         │
│       ▼                                                         │
│  SoACharaDataDic 内部:                                          │
│    1. HashTable で hash → index を解決  (O(1))                  │
│    2. UnsafeList<CharacterBaseInfo>[index] を ref 返し           │
│       │                                                         │
│       ▼                                                         │
│  呼び出し元で ref 経由で直接読み書き                             │
│  ref var info = ref GameManager.Data.GetBaseInfo(hash);         │
│  info.currentHp -= damage;  // 直接書き込み                     │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 AIBrainからのアクセス

AIBrainはSoACharaDataDicのGetXxx(hash)メソッド経由でデータにアクセスする。各AIBrainが自身のhashCodeを保持し、必要なデータを取得してルールベースの判定を行う。

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIBrain アクセスフロー                         │
│                                                                 │
│  AIManager.EvaluateAll()                                        │
│       │                                                         │
│       ▼                                                         │
│  foreach (var brain in _activeBrains)                            │
│  {                                                              │
│      brain.Evaluate();                                          │
│  }                                                              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐    │
│  │ AIBrain.Evaluate()                                       │    │
│  │ {                                                        │    │
│  │     // hash経由でデータアクセス                           │    │
│  │     ref var baseInfo = ref                                │    │
│  │         GameManager.Data.GetBaseInfo(_objectHash);        │    │
│  │     ref var recognition = ref                             │    │
│  │         GameManager.Data.GetRecognition(_objectHash);     │    │
│  │                                                          │    │
│  │     // ルールベースAI判定ロジック...                      │    │
│  │                                                          │    │
│  │     GameManager.Data.SetMovementInfo(                     │    │
│  │         _objectHash, result);  // 結果書き込み           │    │
│  │ }                                                        │    │
│  └─────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────┘
```

## 8. 既存実装からの移行方針

現在の実装（BrainStatus, AIManager, SoACharaDataDic 等）は既に高品質な基盤がある。本設計では以下を**追加・拡張**する。

### 8.1 移行対応表

| 既存実装 | 設計での変更 | 優先度 |
|---|---|---|
| AIManager がコンテナを直接保持 | → **GameManager** がコンテナを保持。AIManager はアクセスのみ | 高 |
| BaseController | → **BaseCharacter** に改名。ObjectHash保持。Ability管理追加 | 高 |
| BrainStatus に混在していた情報 | → **CharacterInfo** / **AttackInfo** / **AIInfo** に分離 | 高 |
| 各所に散在するシングルトン参照 | → GameManager.Instance / GameManager.Data に統一 | 高 |
| GetComponentによるデータアクセス | → objectHash + GameManager.Data に置換 | 中 |
| ダメージ計算（未実装） | → **DamageReceiver** / **DamageDealer** を新規追加 | 中 |
| 武器システム（未実装） | → **WeaponHolder** / **IWeaponBehaviour** を新規追加 | 中 |
| Ability系（未実装） | → **IAbility** + **AbilityBase** + 各アビリティを新規追加 | 中 |
| SoACharaDataDic | → SourceGenerator の属性定義を拡張（StaminaInfo等追加） | 低 |

### 8.2 移行ステップ

```
Phase 1: GameManager導入 ─────────────────────────────────────────
  ① GameManager シングルトンの作成
  ② SoACharaDataDic の所有権を AIManager → GameManager に移動
  ③ AIManager を GameManager 配下に配置
  ④ 既存コードの参照先を GameManager 経由に変更

Phase 2: BaseCharacter整備 ───────────────────────────────────────
  ① BaseController → BaseCharacter に改名
  ② ObjectHash / CharIndex の管理をBaseCharacterに追加
  ③ GameManager.RegisterCharacter() 呼び出しをInitializeに追加
  ④ IAbility インターフェースと AbilityBase 基底クラスの作成

Phase 3: 情報クラス分離 ──────────────────────────────────────────
  ① BrainStatus → CharacterInfo + AttackInfo + AIInfo に分離
  ② SourceGenerator の属性定義を新情報クラスに対応させる
  ③ SoACharaDataDic の自動生成コードを更新

Phase 4: 戦闘システム追加 ────────────────────────────────────────
  ① WeaponHolder / IWeaponBehaviour の実装
  ② DamageDealer / DamageReceiver の実装
  ③ 多属性ダメージ計算の実装
  ④ パリィ・ガードシステムの実装
```

### 8.3 移行時の注意点

```
┌──────────────────────────────────────────────────────────────────┐
│ 移行時の重要な注意点                                              │
├──────────────────────────────────────────────────────────────────┤
│                                                                    │
│  1. AIManager の既存機能を壊さない                                │
│     → SoACharaDataDic の所有権移動時、AIManager の                │
│       EvaluateAll() は GameManager.Data 経由でアクセスするよう変更 │
│                                                                    │
│  2. SourceGenerator の生成コードとの整合性                         │
│     → 情報クラス分離時、[SoAField] 属性の付け方が変わる           │
│     → 生成コードのテストを必ず実行                                 │
│                                                                    │
│  3. AIBrain のデータ安全性                                         │
│     → GameManager.Data への読み書きと AIBrain の評価が              │
│       競合しないよう、実行順序を正しく管理                          │
│                                                                    │
│  4. 段階的移行                                                     │
│     → 一度に全部変更せず、Phase ごとに動作確認                     │
│     → 既存の AIManager + AIBrain のテストが常にパスすること        │
│                                                                    │
└──────────────────────────────────────────────────────────────────┘
```

## 9. 用語集

| 用語 | 説明 |
|---|---|
| **GameManager** | ゲーム全体の中央管理シングルトン。全マネージャーとデータコンテナへの唯一のアクセスポイント |
| **GameManager.Data** | SoACharaDataDic への静的ショートカットプロパティ。`GameManager.Data.GetBaseInfo(hash)` のように使用 |
| **objectHash** | GameObject.GetHashCode() で取得するハッシュコード。GameManager.Dataへのアクセスキーとして使用 |
| **charIndex** | SoACharaDataDic 内でのキャラクターデータの配列インデックス。内部的な直接アクセスに使用 |
| **SoA** | Structure of Arrays。データをフィールドごとの配列で管理するパターン。キャッシュ効率が高い |
| **SourceGenerator** | C#コンパイル時コード生成。[SoAField]属性が付いた構造体からSoAコンテナのボイラープレートを自動生成 |
| **SoACharaDataDic** | SourceGeneratorにより自動生成されるSoAデータコンテナ。全キャラのデータを配列で管理 |
| **UnsafeList\<T\>** | Unity Collections の非マネージドリスト。SoACharaDataDicの内部データ格納に使用 |
| **AIBrain** | MonoBehaviour ベースのルールベースAI判定コンポーネント。SoACharaDataDic経由でデータにアクセスし行動を決定 |
| **MovementInfo** | AIBrain の出力構造体。ターゲット・行動番号・移動方向等のAI判定結果を格納 |
| **RecognitionData** | SensorSystem が収集した知覚・認識データ。周囲の敵・味方・オブジェクト情報 |
| **BaseCharacter** | 各キャラクターのメインMonoBehaviour。ObjectHashとCharIndexを保持し、Ability・Weapon・Damageを統合管理 |
| **IAbility** | アクションコンポーネントのインターフェース。ジャンプ、ダッシュ等の行動を定義 |
| **AbilityBase** | IAbility の基底実装。共通処理（オーナー参照、ハッシュ保持等）を提供 |
| **CharacterInfo** | キャラクターの固有情報を管理するScriptableObject。CharacterBaseInfo、SolidData、MoveStatus等を含む |
| **AttackInfo** | 攻撃・魔法の全情報を管理するScriptableObject。モーション、エフェクト、飛翔体、状態異常等を定義 |
| **AIInfo** | AI の判定ルール・モード・クールタイムを管理するScriptableObject。既存BrainStatus相当 |
| **WeaponHolder** | 武器の装備・切り替え・コンボ・チャージを管理するコンポーネント |
| **DamageDealer** | 攻撃判定・ダメージ発信を担当するコンポーネント。AttackInfo参照からDamageDataを生成 |
| **DamageReceiver** | 被弾時のダメージ計算・パリィ・ガード処理を担当するコンポーネント |
| **DamageData** | 攻撃側から被弾側に渡されるダメージ情報構造体。属性別ダメージ、状態異常、吹き飛ばし等を含む |
| **GetXxxByHash** | SoACharaDataDic のハッシュベースアクセサメソッド群。GetBaseInfo(hash)等でデータにアクセス |
| **PersonalityData** | LLM由来の性格ベクトル(10D)。aggression, survivalPriority等。AIBrain入力に直接含まれる |
| **AnimationStateData** | アニメーションフェーズ（Neutral/Anticipation/Active/Recovery）とキャンセル可否情報 |
| **DeliberationBuffer** | 人間的反応遅延バッファ。AIBrainの出力を遅延させて実行 |
| **ActType.FeintCancel** | Anticipationフェーズ中の攻撃キャンセル。フェイントによる心理戦を創出 |

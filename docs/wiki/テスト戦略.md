# テスト戦略

## テスト種別

| 種別 | 配置先 | 用途 |
|------|--------|------|
| EditMode単体テスト | `Tests/EditMode/[機能名]Tests.cs` | ロジック単体テスト |
| EditMode結合テスト | `Tests/EditMode/Integration_[名前]Tests.cs` | クロスシステムテスト |
| PlayModeテスト | `Tests/PlayMode/[機能名]PlayTests.cs` | MonoBehaviour・物理・シーン |

## テスト命名規則

```
[機能名]_[条件]_[期待結果]
```

例:
```csharp
[Test]
public void DamageCalculation_WhenDefenseIsZero_ShouldDealFullDamage()
{
    // Arrange, Act, Assert
}
```

## EditMode vs PlayMode 使い分け

| EditMode | PlayMode |
|----------|----------|
| 純粋ロジック・計算 | MonoBehaviour連携 |
| データ変換・バリデーション | 物理演算・コライダー |
| 状態遷移ロジック | コルーチン・async |
| AI判定・条件評価 | シーンロード・遷移 |

## 結合テスト設計

### 必須観点

1. **既存ロジック呼び出し検証**
   - 新コードが既存ユーティリティを正しく経由しているか
   - 例: ProjectileHitProcessorがHpArmorLogic.ApplyDamageを経由し、HPクランプ・アーマー処理が適用されるか

2. **状態シーケンス検証**
   - 同じ操作を複数回行った時に状態が壊れないか
   - 例: ActionExecutorでA→B→Aと実行後、OnCompletedが1回だけ発火するか

3. **境界値・不変条件検証**
   - HP < 0 にならない、インデックス範囲外にならない
   - イベント購読数が増え続けない（subscribe/unsubscribeの対称性）

### 機能実装時チェックリスト

- [ ] 他システムのメソッドを呼んでいるか？ → 呼び先の効果まで検証
- [ ] イベントを購読/発行しているか？ → 購読解除・多重購読テスト
- [ ] 状態を持つか？ → 連続操作・リセット後の再操作テスト
- [ ] リソースを確保するか？ → 解放テスト

## テスト実行

```bash
# Unity CLI
Unity.exe -runTests -testPlatform EditMode -projectPath .
Unity.exe -runTests -testPlatform PlayMode -projectPath .

# Claude Code スキル
/run-tests
```
